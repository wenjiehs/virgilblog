## 一、需求背景

### 1.1 背景描述

随着企业数字化转型加速，越来越多的企业采用容器化技术（如 TKE）来管理云原生应用。然而，传统的运维方式面临三大挑战：

1. **运维复杂度高**：SRE 需要掌握 Kubernetes、监控、日志分析等多种技能，学习曲线陡峭
2. **工具碎片化**：企业内部存在多个运维工具（监控系统、发布平台、配置中心），切换成本高
3. **协作效率低**：开发、测试、运维团队使用不同的工具，信息传递存在断层

**OpenClaw** 作为新一代 AI 运维助手，能够通过自然语言理解用户意图，自动执行运维任务（如扩容、日志分析、故障排查），显著降低运维门槛。但企业在部署 OpenClaw 时面临新的问题：

- **部署困难**：需要懂 Kubernetes、Helm Chart、微服务架构才能部署
- **运维负担**：每个团队独立部署，导致版本不一致、安全配置参差不齐
- **成本失控**：缺乏统一的资源管理和成本监控，容易造成资源浪费
- **安全合规风险**：多租户场景下，数据隔离和权限管理复杂

### 1.2 问题定义

**核心问题**：企业如何让数百名员工（SRE、开发者、测试工程师）快速、安全、低成本地使用 OpenClaw？

**关键矛盾**：
- **员工视角**："我只想要一个能用的 OpenClaw，不想学 K8s"
- **IT 管理员视角**："我要保证安全合规、成本可控、版本统一，不能让员工随意部署"

### 1.3 业务目标

#### 短期目标（6 个月）
1. **产品验证**：完成 MVP 版本，支持 10 家企业 POC（每家 5-10 个用户）
2. **商业化启动**：获得 50 家付费客户，ARR 达到 ¥500 万
3. **技术验证**：证明 TKE 上的自动化部署能力，实现 5 分钟内创建实例

#### 长期目标（12 个月）
1. **规模化增长**：200 家企业客户，5000 个活跃实例，ARR ¥3000 万
2. **生态建设**：建立 Skill 市场，沉淀 100+ 企业级运维 Skills
3. **行业标准**：成为 "AI 运维助手 SaaS 化" 的行业标杆产品

#### 业务价值
- **对企业客户**：降低 80% 的部署和运维成本，提升 3 倍运维效率
- **对 TKE 平台**：增加 TKE 的使用粘性，带动容器化业务增长
- **对腾讯云**：新的 SaaS 收入来源，年收入潜力 ¥1 亿+

## 三、需求概述

### 3.1 需求描述

**产品定位**：OpenClaw-as-a-Service（OaaS） —— 企业级 AI 运维助手 SaaS 管理平台

**一句话描述**：让企业员工像申请云主机一样，一键获取属于自己的 OpenClaw AI 运维助手，IT 管理员可统一管理、监控和控制成本。

**核心能力**：
1. **自助申请**：员工填写简单表单，5 分钟获得可用的 OpenClaw 实例
2. **自动部署**：系统自动在 TKE 集群创建独立的 Namespace、Pod、PVC、Ingress
3. **统一管理**：管理员一个控制台管理所有实例（审批、监控、升级、成本）
4. **安全隔离**：每个员工的实例完全隔离（Namespace + RBAC + NetworkPolicy）
5. **成本可控**：按实例规格计费，支持配额管理和成本分摊

### 3.2 核心价值

#### 对员工（使用者）
- **降低使用门槛**：无需懂 K8s，只需填写表单即可获得 OpenClaw
- **快速交付**：从申请到使用 < 30 分钟（传统方式需要 2-3 天）
- **开箱即用**：自动配置企业微信/Slack 集成，内置企业常用 Skills
- **成本透明**：实时查看自己的使用量和费用

#### 对 IT 管理员（运营者）
- **减少运维工作量**：从人工部署变为自动化，节省 90% 时间
- **全局可控**：一个大盘监控所有实例，异常自动告警
- **成本可管理**：按部门分摊费用，设置配额防止滥用
- **安全合规**：自动记录审计日志，满足等保要求

#### 对企业（决策者）
- **提升运维效率**：员工使用 AI 助手后，运维效率提升 3 倍
- **降低 IT 成本**：减少运维团队的重复性工作，人力成本降低 50%
- **统一工具体系**：避免工具碎片化，降低培训成本

### 3.3 成功指标（北极星指标）

#### 一级指标：活跃实例数（North Star Metric）
- **定义**：每周至少有 1 次 API 调用的实例数量
- **目标**：
  - MVP 阶段（3 个月）：50 个活跃实例
  - 产品化阶段（6 个月）：500 个活跃实例
  - 规模化阶段（12 个月）：5000 个活跃实例

#### 二级指标（关键过程指标）

| 指标类别 | 指标名称 | 计算方式 | 目标值（12 个月） |
|---------|---------|---------|-----------------|
| **用户增长** | 企业客户数 | 付费企业数量 | 200 家 |
| **用户增长** | 用户留存率 | 30 天后仍活跃的用户比例 | ≥ 70% |
| **产品体验** | 申请到使用时长 | 从提交申请到实例可用的中位数时间 | < 15 分钟 |
| **产品体验** | 用户满意度（NPS） | Net Promoter Score | ≥ 50 |
| **运营效率** | 自动化部署成功率 | 部署成功次数 / 总部署次数 | ≥ 99% |
| **运营效率** | 管理员操作时长 | 管理员每周花费在平台上的时间 | < 2 小时/周 |
| **商业化** | ARR（年度经常性收入） | 所有付费客户的年化收入 | ¥3000 万 |
| **商业化** | 客单价 | ARR / 企业客户数 | ¥15 万/年 |
| **技术稳定性** | 平台可用性（SLA） | 正常运行时间 / 总时间 | 99.9% |
| **技术稳定性** | P0 故障次数 | 导致所有实例不可用的故障 | < 1 次/季度 |

### 4.2 核心功能详细说明

#### 功能 1：实例申请流程（员工端）

**功能目标**：让员工通过简单的表单，快速申请一个可用的 OpenClaw 实例

**前置条件**：
- 员工已通过 SSO 登录平台
- 员工所在部门有可用配额（剩余实例数 > 0）

**操作流程**：

```
步骤 1：进入申请页面
├─ 页面元素：
│  ├─ 标题："申请你的 OpenClaw AI 助手"
│  ├─ 说明："5 分钟内获得专属 AI 运维助手，支持自然语言执行运维任务"
│  └─ 按钮："开始申请"

步骤 2：填写申请表单
├─ 必填字段：
│  ├─ [实例名称] 
│  │  ├─ 输入框（自动填充：openclaw-{用户名}）
│  │  ├─ 校验规则：仅支持小写字母、数字、连字符，长度 4-30
│  │  └─ 提示："实例名称将作为访问域名的一部分"
│  │
│  ├─ [用途说明]
│  │  ├─ 文本框（多行，最多 200 字）
│  │  ├─ 占位符示例："用于 TKE 集群日常运维，包括扩容、日志分析、故障排查"
│  │  └─ 提示："详细说明有助于加快审批"
│  │
│  ├─ [实例规格]
│  │  ├─ 单选卡片（可视化对比）：
│  │  │  ├─ ⚪ 基础版（1C2G）- ¥200/月
│  │  │  │  ├─ 适用场景：轻度使用，每天 < 50 次调用
│  │  │  │  └─ 免费额度：1000 次 API 调用/月
│  │  │  ├─ ⚪ 标准版（2C4G）- ¥500/月【推荐】
│  │  │  │  ├─ 适用场景：日常运维，每天 50-200 次调用
│  │  │  │  └─ 免费额度：5000 次 API 调用/月
│  │  │  └─ ⚪ 企业版（4C8G）- ¥1000/月
│  │  │     ├─ 适用场景：重度使用，每天 > 200 次调用
│  │  │     └─ 免费额度：20000 次 API 调用/月
│  │  └─ 提示："不确定选哪个？先选标准版，后续可升级"
│  │
│  └─ [使用时长]
│     ├─ 单选：
│     │  ├─ ⚪ 长期（无固定期限）
│     │  └─ ⚪ 临时（7 天后自动删除）
│     └─ 提示："临时实例适用于测试场景，数据不保留"
│
├─ 选填字段：
│  └─ [企业微信集成]
│     ├─ 开关：是否配置企业微信 Bot
│     └─ 说明："开启后可直接在企业微信中使用 OpenClaw"
│
└─ 底部信息面板（自动计算）：
   ├─ 预估费用：¥500/月（标准版）
   ├─ 当前配额：你的部门剩余 15/20 个实例
   └─ 预计交付时间：审批通过后 5 分钟内

步骤 3：提交申请
├─ 点击"提交申请"按钮
├─ 前端校验：
│  ├─ 检查必填字段
│  └─ 检查实例名称是否已被占用
├─ 后端处理：
│  ├─ 创建申请记录（状态：待审批）
│  ├─ 自动流转到部门管理员
│  └─ 发送企业微信通知给管理员
└─ 页面跳转：申请提交成功页面

步骤 4：等待审批
├─ 申请提交成功页面：
│  ├─ 提示："你的申请已提交，等待管理员审批"
│  ├─ 申请单号：#20260304001
│  ├─ 审批人：李经理（IT 运维经理）
│  └─ 预计审批时间：< 1 小时
├─ "我的实例"列表显示：
│  ├─ 实例名称：openclaw-zhangsan
│  ├─ 状态：🟡 待审批
│  └─ 操作：[撤回申请]
└─ 审批通过后：
   ├─ 企业微信通知："你的 OpenClaw 申请已通过，正在部署..."
   └─ 邮件通知：包含详细信息

步骤 5：开始使用
└─ 企业微信通知（包含以下信息）：
   ├─ 标题："你的 OpenClaw 已就绪！🎉"
   ├─ 访问地址：https://openclaw-zhangsan.tke-cloud.com
   ├─ 企业微信 Bot：已自动配置，直接在企业微信里 @OpenClaw 即可
   ├─ 快速入门：[3 分钟视频教程]
   └─ 示例命令："帮我查看 production 命名空间下所有 Pod"
```

**业务规则**：

| 规则类型 | 规则内容 |
|---------|---------|
| **配额检查** | 提交申请前，检查用户所在部门的配额；配额不足时禁止提交，提示"请联系管理员增加配额" |
| **实例名称唯一性** | 全局唯一，不允许重复；建议格式：`openclaw-{用户名}` 或 `openclaw-{用户名}-{用途}` |
| **自动审批规则** | 如果用户申请"基础版"且部门配额充足，可配置自动审批（跳过人工审批） |
| **临时实例回收** | 临时实例到期前 24 小时发送提醒；到期后自动删除，数据不保留 |
| **申请撤回** | 审批通过前可撤回申请；审批通过后不允许撤回，只能删除实例 |

**异常处理**：

| 异常场景 | 处理方式 |
|---------|---------|
| 部署失败（TKE API 错误） | 自动重试 3 次；仍失败则标记为"部署失败"，通知管理员人工介入 |
| 实例名称冲突 | 前端提示："实例名称已存在，请更换" |
| 配额不足 | 前端提示："你的部门配额已用完，请联系管理员：李经理（企业微信：xxx）" |
| 审批超时（24 小时未审批） | 自动发送催办通知给管理员；48 小时仍未审批则自动拒绝 |

#### 功能 2：管理员审批工作台

**功能目标**：让管理员高效审批申请，快速了解申请的合理性

**前置条件**：
- 用户具有管理员角色
- 存在待审批的申请

**操作流程**：

```
步骤 1：查看待审批列表
├─ 页面元素：
│  ├─ 标题："待审批申请"
│  ├─ 过滤器：
│  │  ├─ 部门：[全部 ▼]  [研发部]  [测试部]  [产品部]
│  │  ├─ 规格：[全部 ▼]  [基础版]  [标准版]  [企业版]
│  │  └─ 排序：[申请时间 ▼]  [部门]  [规格]
│  └─ 批量操作：[批量批准]  [批量拒绝]
│
└─ 申请列表（表格形式）

步骤 2：查看申请详情
├─ 点击"详情"按钮 → 弹出抽屉面板：
│  ├─ 申请信息：申请人、部门、申请时间、实例名称
│  ├─ 资源信息：规格、使用时长、用途说明
│  ├─ 配额检查：部门配额、个人配额、集群资源
│  ├─ 成本预估：月费用、年化成本
│  └─ 历史记录：申请人过去的申请记录

步骤 3：审批决策
├─ 批准流程：
│  ├─ 点击"批准"按钮
│  ├─ 可选：填写批准备注
│  ├─ 确认批准 → 后台自动触发部署
│  └─ 发送通知给申请人
│
└─ 拒绝流程：
   ├─ 点击"拒绝"按钮
   ├─ 必填：拒绝原因
   ├─ 确认拒绝 → 申请状态变更为"已拒绝"
   └─ 发送通知给申请人
```

#### 功能 3：全局监控大盘（管理员端）

**功能目标**：让管理员一屏掌握所有实例的运行状态、成本和异常

**核心指标卡片**：
- 总实例数 / 配额
- 活跃实例数（7 个休眠）
- 本月成本 / 预算
- 异常告警数量

**趋势图表**：
- 实例数趋势（近 30 天）
- 成本趋势（近 30 天）

**异常告警列表**：
- 实例名称、异常原因、建议操作
- 快捷操作按钮：查看详情、重启实例、升级规格

**部门资源使用情况**：
- 各部门的实例数、配额、本月成本、占比
