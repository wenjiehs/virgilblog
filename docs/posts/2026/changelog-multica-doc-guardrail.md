# 把 Changelog 变成文档守门人：我用 Multica 串起产品发布和官网文档

## 🤔 为什么又折腾了一套流程

最近我在做一个很具体的事情：**把研发提交记录自动整理成产品 Changelog**。

一开始目标很简单：每周从核心仓库的 `master` 分支拉取合入记录，按产品视角整理成两份内容：

- **内部版**：保留 commit、MR、作者、原始标题，方便研发和产品追溯。
- **对外版**：去掉内部信息，转成客户能看懂的产品更新说明。

这个方向很快就跑通了。AI 能根据规则把一堆 commit 标题整理成“新增功能”“体验优化”“稳定性修复”“兼容性更新”等分类，质量比我预期好。

但跑了几期之后，我发现一个更关键的问题：

> **Changelog 只告诉我们“发布了什么”，但没有保障“文档是否跟上了”。**

这在产品工作里其实很常见：

- 新功能上线了，但官网文档没更新。
- 控制台入口变了，但操作指南还是旧截图。
- 支持了新机型、新镜像、新地域，但使用限制里没有体现。
- 对外公告写了“新增支持”，客户点进文档却找不到怎么用。

所以我想把 Changelog 再往前推一步：不只是生成公告，而是变成一个“文档守门人”。

---

## 🧩 原来的 Changelog 项目做了什么

这个 Changelog 项目本质上是一个“研发语言 → 产品语言”的翻译器。

输入是内部代码仓库里的提交记录，输出是结构化的产品更新说明。

简化后的流程是：

```text
master 分支提交记录
    ↓
提取 first-parent commit / MR 信息
    ↓
按规则分类和过滤
    ↓
生成内部版 Changelog
    ↓
生成对外版 Changelog
    ↓
保存到内部归档仓库
```

为什么要看 `first-parent`？因为实际仓库里很多合入不是传统 merge commit，而是 squash 或 fast-forward 风格，提交标题里带有 MR 信息。如果只看 `git log --merges`，会漏掉大量真实发布内容。

项目里沉淀了几类规则：

| 类型 | 处理方式 |
|---|---|
| `feat` / 产品需求 | 优先进入对外版 |
| `fix` | 转成“稳定性提升”或“问题修复” |
| `policy` / 镜像 / 机型 / 地域 | 合并为“支持范围更新” |
| `test` / `chore` / `docs` | 默认只进内部版，不对外发布 |
| 内部实现细节 | 对外版必须删除 |

这个项目最有价值的地方，不是“自动写几段文字”，而是把判断逻辑固定下来了：**什么能对外讲，什么只能内部看，什么需要合并表达。**

---

## 🔗 为什么要和 Multica 结合

本地跑脚本能解决“我自己生成”的问题，但解决不了团队协作。

我希望它变成一个长期运行的产品工作流：

```text
每周自动触发
    ↓
生成 Changelog
    ↓
创建 Issue 给团队 review
    ↓
结果回写归档仓库
    ↓
继续触发文档更新和文档覆盖检查
```

这就很适合放到 Multica 里。

Multica 的几个概念刚好能对应上：

| Multica 组件 | 在这个流程里的作用 |
|---|---|
| Agent | 执行 Changelog 生成、文档同步、文档检查 |
| Issue | 每次周更的任务和讨论载体 |
| Label | 区分 changelog、doc、review 等任务类型 |
| Project | 把相关 Issue 收拢到同一个看板 |
| Autopilot | 每周定时触发 |

我最后采用的不是“一个 Agent 干所有事”，而是拆成几条链路。

---

## 🏗️ 第一条链路：自动生成 Changelog

第一条链路很直接：每周生成一期 Changelog。

```text
每周 Autopilot
    ↓
Changelog Agent
    ↓
拉取内部代码仓库提交记录
    ↓
生成 internal-changelog.md / public-changelog.md
    ↓
commit & push 回归档仓库
    ↓
在 Issue 里评论摘要和风险点
```

这里我特意要求“双写”：

1. **写回归档仓库**：作为长期记录和版本追溯。
2. **写到 Issue 评论**：方便产品、研发、测试在同一个地方 review。

这两个用途不一样。仓库适合沉淀，Issue 适合协作。

Issue 里不需要贴所有内部细节，只要包含：

- 本次数据范围
- 对外版摘要
- 完整文件位置
- 需要人工确认的风险点

比如：某个能力是否已经全量发布，某个术语是否适合对外表达，某个组件版本是否只是灰度。

---

## 📣 第二条链路：同步到产品更新历史

有了对外版 Changelog 后，下一步就是把它变成官网里的“产品更新历史”。

这条链路我没有放在 Changelog Agent 里，而是单独做了一个文档同步 Agent。

```text
Changelog Issue 生成完成
    ↓
文档同步 Autopilot 定时触发
    ↓
创建“更新产品历史文档”的 Issue
    ↓
文档同步 Agent 读取最新对外版 Changelog
    ↓
调用文档写作能力
    ↓
更新产品更新历史文档草稿
```

为什么不直接让 Changelog Agent 改官网文档？

因为职责不一样。

Changelog Agent 更像“产品发布记录整理员”，它擅长从 commit 里提炼产品语言；文档同步 Agent 更像“官网文档编辑”，它需要遵守官网写作规范、处理文档系统的读写、避免破坏已有内容。

这两个角色拆开之后，指令更清晰，风险也低很多。

---

## 🧭 第三条链路：检查功能有没有对应文档

这是我觉得最有价值的一步。

对外公告只是告诉用户“我们做了什么”，但用户真正需要的是“我该怎么用”。

所以每个对外发布的功能，都应该过一遍文档覆盖检查。

设计流程是：

```text
读取本期 public-changelog.md
    ↓
抽取对外发布功能项
    ↓
读取官网文档目录树
    ↓
逐条匹配是否已有对应文档
    ↓
输出覆盖检查报告
    ↓
缺文档的功能自动创建文档 Issue
    ↓
交给文档撰写 Agent 处理
```

我把检查结果分成四类：

| 状态 | 含义 | 后续动作 |
|---|---|---|
| 已覆盖 | 已有明确文档承接 | 可只更新产品历史 |
| 可能需要更新 | 有相关文档，但不确定是否覆盖新能力 | 创建更新建议 |
| 缺少文档 | 找不到明显对应文档 | 创建文档补充 Issue |
| 暂不处理 | 稳定性修复、内部变更、灰度能力 | 仅记录或过滤 |

同时给每个条目标优先级：

- **P0**：新增用户可操作功能，发布前必须有文档。
- **P1**：已有功能增强，建议本周补齐。
- **P2**：稳定性修复或兼容性优化，可只进产品动态。

举个例子：

```text
节点池支持添加已有节点
    → P0
    → 必须检查是否有"添加已有节点"的操作指南

Ingress 支持按规则域名搜索
    → P1
    → 大概率更新已有 Ingress 管理文档即可

修复某个资源泄漏问题
    → P2
    → 写进产品更新历史即可，不一定需要单独文档
```

这样一来，Changelog 不只是"发布记录"，而变成了"文档完整性的扫描器"。

### 资源清单

| 类型 | 名称 | 职责 |
|------|------|------|
| Agent | Changelog Bot | 从代码仓库 master 生成 changelog，双写归档 + Issue |
| Agent | 文档同步员 | 把对外版 changelog 写入产品更新历史文档 |
| Agent | 文档覆盖检查员 | 检查功能是否有文档，缺则建 Issue 并带 MR 链接 |
| Agent | 文档撰写 Agent | 兜底处理 doc Issue，生成文档草稿 |
| Autopilot | Changelog 周更 | 周一 09:00 触发 Changelog Bot |
| Autopilot | 产品更新历史同步 | 周一 11:00 触发文档同步员 |
| Autopilot | 功能文档覆盖检查 | 周一 11:00 触发覆盖检查员 |

| 项目 | 文档撰写 | 所有 doc Issue |
| 标签 | `changelog` | 标记 Changelog 相关 Issue |
| 标签 | `doc` | 标记文档类 Issue |
| 仓库 | Changelog 归档 | `outputs/` 保存历史、`state/` 保存增量游标 |

### 五条核心设计原则

把整个系统的设计提炼成五句话：

1. **代码变更驱动文档** —— 不靠人记忆，每次 master 合入都自动追踪。
2. **MR 是第一手资料** —— doc Issue 直接挂 MR 链接，撰写者看代码 diff 写文档。
3. **谁开发谁负责文档** —— Issue assign 给 MR 作者，而不是统一丢给文档团队。
4. **双写归档** —— 仓库留历史，Issue 留协作，文档系统发布对外。
5. **自动化 + 人工审核** —— 全流程自动触发，最后一步人工 review 才发布。

这五条规则其实是这套系统能跑通的关键。技术上的事情（Agent、Autopilot、标签）只是载体，规则才是骨架。

## 🧱 为什么最后用了三个 Autopilot

我一开始想过：要不要让一个 Agent 串完整个流程？

后来还是拆成了三个 Autopilot：

| 时间 | Autopilot | 职责 |
|---|---|---|
| 周一上午 | Changelog 周更 | 生成本周 Changelog，并写回归档仓库 |
| 周一稍晚 | 产品更新历史同步 | 把对外版 Changelog 写入产品更新历史 |
| 周一稍晚 | 功能文档覆盖检查 | 检查新功能是否有文档，没有就提 Issue |

拆开的好处是：

1. 每条链路失败时，影响范围小。
2. Issue 更清晰，每个任务有独立上下文。
3. 后续想调整某条链路，不会影响其他链路。
4. 文档相关任务可以交给专门的文档 Agent。

缺点也有：

- 调度时间需要错开。
- 如果 Changelog 延迟生成，后两个任务可能读不到最新结果。
- 首次配置比一个大 Agent 麻烦。

但我还是更倾向这种方式。它更像真实团队里的分工：发布记录、官网公告、文档检查，本来就不是同一件事。

## 🕰️ 一个容易忽略的问题：存量数据怎么办

搭完增量流程后，我发现一个问题：

> 三个 Autopilot 只能覆盖未来每周的新增 Changelog，覆盖不了从年初到现在已经生成的存量内容。

这其实挺典型的。自动化系统刚上线时，经常只解决“从今天开始”的问题，但历史数据没人消费。

所以我又补了两个一次性任务：

### 1. 存量 Changelog 同步到产品更新历史

让文档同步 Agent 按时间顺序读取过去几个月的所有对外版 Changelog，一次性补录到产品更新历史文档里。

### 2. 存量功能文档覆盖检查

让覆盖检查 Agent 扫描所有历史对外版 Changelog，把里面的功能项全部过一遍。如果发现缺文档，就创建文档补充 Issue。

这样做完之后，系统才算真正进入“干净状态”：

```text
历史数据：一次性补录和检查
未来增量：每周 Autopilot 自动处理
```

这个教训挺重要：**做自动化时，不仅要设计增量流程，还要设计存量迁移。**

## 🪤 踩到的几个坑

### 1. Issue 不是归档仓库的替代品

一开始我差点只把结果发到 Issue。后来想想不对，Issue 适合讨论，但不适合做长期稳定归档。

所以最后变成：

- 仓库存完整 Markdown。
- Issue 放摘要、链接和讨论。

这个分工更稳。

### 2. 对外版不能直接复制 commit 语言

很多 commit 标题对研发很清楚，但对客户很奇怪。

比如“修复 nil panic”“改用某接口”“增加字段”这类表达，都不适合直接写到对外公告。需要改写成：

- 修复特殊配置下的异常问题。
- 优化系统盘扩容询价体验。
- 增强集群类型识别能力。

这也是 AI 最适合发挥作用的地方：把研发语言翻译成产品语言。

### 3. 文档覆盖检查不能太激进

不是每条 Changelog 都要新建文档。

如果每个修复都提一个文档 Issue，文档团队会被淹没。所以需要明确规则：新增功能和用户操作变化优先，稳定性修复多数只进入产品动态。

### 4. Agent 之间最好通过 Issue 协作，而不是互相直接调用

Issue 有几个好处：

- 有上下文。
- 有评论历史。
- 可以人工介入。
- 可以打标签和归项目。
- 可以失败后重跑。

这比一个 Agent 在后台默默串调用更透明。

---

## 💭 我对这件事的理解

这次实践让我对“AI Agent 做产品工作”有了一个更具体的感受。

过去我们常说 AI 能帮产品经理写文档、写公告、写周报。但如果只是“给它一段材料，让它生成一段文字”，价值其实有限。

真正有价值的是把它放进流程里：

```text
研发实际交付
    ↓
产品语言总结
    ↓
对外公告
    ↓
官网文档
    ↓
文档缺口 Issue
    ↓
再次进入团队工作流
```

这就不是一次性的内容生成，而是一个闭环。

它解决的也不是“写得快一点”，而是几个更实际的问题：

- 产品经理不用每周手动翻 MR。
- 对外公告有稳定来源。
- 官网产品更新历史不会靠记忆维护。
- 新功能发布前能发现文档缺口。
- 文档任务能自动进入团队协作系统。

当然，这套系统还远谈不上完美。功能项和文档的匹配仍然可能误判；灰度能力是否能对外讲也需要人工确认；产品语言是否准确还要产品经理 review。

但我觉得方向是对的：**让 AI 先把重复、繁琐、容易遗漏的部分接住，人再去做判断和确认。**

## 📌 下一步想做什么

后续我想继续补几件事：

1. **建立功能到文档的映射表**  
   把常见模块和官网文档位置沉淀下来，减少每次靠语义猜。

2. **把覆盖检查报告结构化**  
   每个功能项都输出状态、优先级、建议动作和目标文档。

3. **增加发布前检查清单**  
   对外公告发布前，自动提示“这些功能文档还没补”。

4. **把文档 Issue 和 Changelog 反向关联**  
   以后看到某篇文档更新，可以追溯它来自哪一期 Changelog。

5. **沉淀人工复核规则**  
   哪些词不能对外讲，哪些能力要等全量发布，哪些属于内部实现，都要逐步写进规则。

## 🎉 总结

这次实践表面上是在做 Changelog 自动化，实际上是在尝试一件更大的事：

> **把研发交付自动转化成产品资产。**

代码提交本身不是产品资产，MR 标题也不是。只有经过分类、脱敏、产品化表达、文档承接之后，它才真正变成客户能感知的产品进展。

Multica 在这里扮演的角色，不只是“定时跑一个 Agent”，而是把多个 Agent、Issue、项目、标签和 Autopilot 串成一个可协作的工作流。

对我来说，这比单纯让 AI 写一篇公告更有意思。因为它开始触碰产品工作的核心：**如何让团队已经做出来的东西，被用户正确理解、正确使用。**
