# 从策划稿到功能上线：AI 产品策划工作站的真实生产过程

## 🤔 背景

上一篇文章 [让 AI 学会我的产品，再帮我写策划稿](/posts/2026/ai-product-planning-tcr-skills) 讲的是「搭建工作站」的过程——让 AI 自主学习 TCR 产品逻辑，建立知识库，然后生成策划稿。

那篇文章的核心是**建设阶段**：搭框架、喂知识、调规则。

这篇是**续集**，记录的是**真正用这套方法做产品迭代**的过程。

今天（2026-03-19）我用这个工作站完成了一个真实的产品需求，从发现问题到输出策划稿和交互原型，整个过程大概 **3-4 小时**。

---

## 🎯 今天要做什么

### 背景

TCR 最近上线了 AI 模型管理（modctl CLI）和 AI Skills 管理（clawhub CLI）两个新功能。研发同学给了我一份"命令体验手册"，让我从产品角度评估一下。

### 我的预期

1. **体验评估**：实际执行命令，找出产品问题
2. **用户指南**：写一份面向用户的操作手册
3. **控制台优化**：发现的问题，提出控制台侧的改进方案
4. **交互原型**：让开发同学直接看到目标效果

---

## 🛠️ 实际过程

### 第一步：实测 CLI 工具

我把研发的"命令体验手册"给 AI，让它**实际执行命令**，而不是只看文档。

AI 做了什么：

1. **安装 CLI 工具**
   - clawhub：`npm install -g clawhub` ✅ 成功
   - modctl：Homebrew tap 404、GitHub Release 404，最后从源码编译 ✅

2. **逐条执行命令**
   - 连通性测试（curl ping）
   - 登录验证
   - 发布 Skill
   - 搜索 Skill
   - 安装 Skill
   - ...

3. **记录实际输出**
   - 每条命令的实际返回
   - 成功/失败状态
   - 遇到的问题

**关键发现**：modctl 的安装文档有问题（Homebrew tap 和 GitHub Release 链接都 404），用户根本装不上。这种问题只有实际执行才能发现。

### 第二步：生成体验报告

基于实测结果，AI 自动生成了一份产品体验报告：

**报告结构**：
- 体验对象和测试环境
- 安装情况（表格）
- 实测结果汇总（命令+输出+状态）
- 整体评价（四个维度打分）
- 亮点和待改进点
- 产品建议

**我觉得有意思的是**：AI 会自己总结出"亮点"和"待改进"，而且判断挺准的。比如它说"modctl 安装困难是 P0 问题"，这个优先级判断我认同。

### 第三步：发现控制台问题

在测试登录命令时，我发现一个问题：

```
控制台「生成临时登录指令」只展示 docker login 命令
但用户需要的是 modctl login 命令
```

我截了个图给 AI：

![控制台截图](https://virgil.run/virgilblog/posts/2026/assets/tcr-login-cmd-screenshot.png)

然后问 AI：**"这里的登录指令是前端拼接的，还是后端返回的？"**

AI 分析后认为应该是**后端返回**（因为包含动态生成的 Token），然后主动提出：

> "建议后端返回时同时包含 docker、modctl、clawhub 三种登录命令，前端直接展示。"

这个建议很实用——API 返回结构应该像这样：

```json
{
  "loginCommands": {
    "docker": "docker login xxx --username xxx --password xxx",
    "modctl": "modctl login xxx --username xxx --password xxx",
    "clawhub": "clawhub login xxx --username xxx --password xxx"
  }
}
```

### 第四步：生成策划稿

我让 AI 把这个问题写成一份正式的功能策划稿。

AI 用之前配置好的模板，生成了一份完整的策划稿：

- **背景与动机**：问题描述、用户反馈、核心诉求
- **产品目标**：核心目标 + 成功指标（SMART 原则）
- **功能设计**：当前状态 vs 目标状态（ASCII 图）
- **技术方案**：后端 API 改造 + 前端 UI 改造
- **实施计划**：工时估算 + 验收标准

**我觉得最有用的是 ASCII 图**：

```
┌─────────────────────────────────────────────────────┐
│  📋 登录指令                                         │
├─────────────────────────────────────────────────────┤
│  🐳 Docker（容器镜像）                    [复制]     │
│  docker login xxx --username ...                    │
│                                                     │
│  🤖 modctl（AI 模型管理）                 [复制]     │
│  modctl login xxx --username ...                    │
│                                                     │
│  🔧 clawhub（AI Skills 管理）             [复制]     │
│  clawhub login xxx --username ...                   │
└─────────────────────────────────────────────────────┘
```

这种图直接贴到需求文档里，开发同学一看就懂。

### 第五步：生成交互原型

策划稿写完后，我又提了几个需求：

> "创建一个新的交互原型 HTML 文件，满足：
> - 模型列表页添加「快速入门」引导卡片
> - 详情页添加「CLI 操作指令」区块
> - 添加「上传模型」「下载模型」弹窗
> - Skills 页面同样添加 CLI 指导"

AI 直接生成了一个 **1300+ 行的 HTML 原型**，包含：

1. **快速入门引导卡片**
   - 模型页：安装 CLI → 登录 → 生成 Modelfile → 构建推送
   - Skills 页：安装 CLI → 登录 → 发布/安装

2. **详情页 CLI 操作区块**
   - Tab 切换：拉取/推送/版本管理
   - 代码高亮 + 一键复制

3. **上传/下载弹窗**
   - 完整的步骤指引
   - 每步带序号和代码块
   - 支持一键复制

4. **交互细节**
   - 代码块复制后显示「已复制」
   - 弹窗点击遮罩层关闭
   - 列表页「拉取」按钮打开预填弹窗

AI 还自动启动了本地服务器预览原型。**从提需求到看到可交互的原型，大概 5 分钟**。

### 第六步：更新用户指南

最后，我让 AI 更新用户指南，把今天发现的登录流程细节补充进去：

**更新内容**：
- 区分「用户级账号」和「服务级账号」
- 添加完整的登录流程说明
- 说明 docker/modctl 使用相同的凭证

## 💭 一些感受

### 1. 从"搭建"到"使用"，体验完全不同

上一篇文章讲的是搭建阶段——建规则、训练 AI、积累知识。那个过程其实挺繁琐的。

但今天是**使用阶段**——直接说需求，AI 就能产出。**这才是这套方法的价值所在**。

就像买了一台咖啡机，调试机器很麻烦，但调好之后每天按一个按钮就有咖啡喝。

### 2. AI 实测比纯看文档靠谱

让 AI 实际执行命令，而不是只分析文档，发现的问题更真实。

比如"modctl 安装链接 404"这个问题，看文档看不出来，必须真正去装才知道。

**教训**：做产品体验评估时，一定要让 AI 实际执行，不要只做"文档分析"。

### 3. 截图 + 提问 = 高效沟通

我发现一个高效的工作模式：

1. 截图给 AI
2. 问一个具体问题
3. AI 分析后给建议
4. 我确认后让 AI 输出策划稿

比起"帮我优化这个页面"这种模糊需求，**截图 + 具体问题**让 AI 的输出更精准。

### 4. 交互原型是真的有用

之前写策划稿，描述交互都是用文字，开发同学经常理解偏差。

现在 AI 直接生成 HTML 原型，**可点击、可预览**，沟通效率高很多。

虽然不如专业设计工具精细，但作为"功能原型"完全够用了。

### 5. 工作流在跑，我在旁边"指导"

今天的工作模式是这样的：

```
我：提需求、截图、确认
AI：执行命令、生成报告、写策划稿、做原型
```

我更像是一个"指导者"，而不是"执行者"。AI 做具体的事情，我负责方向和质量把关。

这种感觉挺奇妙的——**产品经理的工作正在被重新定义**。

## 🔧 给其他产品经理的建议

如果你也想试试这套方法，这是我的建议：

### 1. 先投入时间搭建工作站

不要急着让 AI 干活，先花 1-2 天：
- 配置好规则（策划稿模板、评审标准）
- 让 AI 学习你的产品
- 建立知识库

### 2. 让 AI 实际执行，不要只做分析

做产品体验评估时，一定要让 AI **实际执行命令**、**实际操作页面**，而不是只分析文档。

### 3. 截图 + 具体问题

和 AI 沟通时，**截图 + 具体问题**比模糊描述高效得多。

### 4. 让 AI 输出可交互的原型

不要满足于文字描述，让 AI 直接生成 HTML 原型。虽然不完美，但沟通效率高很多。

### 5. 保持人工审核

AI 生成的内容不是直接发出去就行，需要：
- 检查业务逻辑
- 删除过度发挥的内容
- 补充只有你知道的上下文
