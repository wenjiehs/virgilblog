# 用 CLI 重写产品文档：让 AI 时代的文档也能被「编译」

## 🤔 起因

最近我把自己每天的工作做了一次记录，发现一个挺尴尬的事实：

**作为一个云产品的产品经理，我自己也越来越少打开自家产品的控制台。**

为什么？

因为越来越多的事，我会让 AI 帮我做。让 AI 跑 CLI、写脚本、跑流水线，比打开控制台一个个点要快很多。

然后我开始观察身边的开发者朋友、客户的 SRE，他们的趋势是一样的：

- AI 工具普及了，大家越来越习惯说「给我跑一段 CLI 把这个搞定」
- 控制台的角色，逐渐从"主入口"变成"出问题时去看一眼的工具"
- 文档里大量"打开控制台 → 点击 XX → 截图如下"的步骤，**对 AI Agent 完全不可用**

这让我开始认真思考一个问题：

> **AI 时代的云产品文档，应该长什么样？**

这篇文章是我把这件事想清楚之后，给团队写的一份产品策划稿的思考过程。涉及具体产品名、内部链接和敏感数据已做脱敏。

---

## 🎯 我看到的三个核心问题

### 问题 1：CLI 没准备好给 AI 用

主流云厂商都有自家 CLI，但**几乎没有一个是真正为 AI Agent 设计的**。

典型表现：

- 默认输出是给人看的（漂亮表格 + 颜色 + Unicode 边框）
- 参数风格继承 OpenAPI，命名长、必填多
- 命令名风格不统一，动词散乱
- 错误信息是中文/英文混杂的提示，没有可枚举的错误码
- 命令的「能干啥」必须翻文档才知道，无法运行时自省

让 AI 调用这种 CLI，等同于让它"边查文档边写代码"，Token 消耗高、稳定性差。

### 问题 2：文档是为人写的，不是为执行写的

我自己写过、审过大量云产品文档。共同的问题：

- **过度依赖截图**：截图是某一时刻的 UI，UI 一改文档就过时
- **步骤不可执行**：「点击右上角的 XX 按钮」对 Agent 没意义
- **没有结构化**：参数、API、配置、命令各写一段，没法被工具消费
- **缺乏验证机制**：文档写完没人验证它能不能跑通

结果就是 —— **文档腐烂的速度比产品迭代速度还快**。

我做过一个简单统计：在我熟悉的某个云产品文档库里，随机抽查 50 篇操作类文档，**接近 30% 的步骤实际跑不通**（截图过时、入口改名、API 字段变更、产品下线但文档还在）。

### 问题 3：文档没有「编译」

代码有 lint、有编译、有单元测试、有 CI。

文档呢？基本没有。

- 文档里写了一个 CLI 命令，没人保证它在最新版本能跑
- 文档里写了一个 API 调用，没人保证签名和参数是最新的
- 文档里贴了一张控制台截图，没人保证它跟现在的 UI 一致

我们对代码有几百条 Lint 规则，对文档却只有「看起来差不多就行」。

---

## 💡 我提出的解法：三件事一起做

经过反复思考，我觉得这三件事是同一件事，必须一起做：

```text
1. 让 CLI 变成 AI Agent 友好的形态
       ↓
2. 用 CLI 作为骨架重写产品文档
       ↓
3. 给文档加上「编译」能力
```

任何一件单独做都意义有限：

- 只做 CLI，文档还在依赖截图，AI Agent 学不会
- 只做文档改造，CLI 本身不支持，文档里的 CLI 也跑不起来
- 只做文档 CI，没有可执行单元，CI 也只能查错别字

---

## 🛠️ 第一步：Agent-Native CLI

CLI 真正为 AI Agent 设计，意味着要重新建立一套设计原则。

我把它整理成一张表：

| 原则 | 说明 |
|------|------|
| **结构化输出** | 默认 `--output json`，不能给 Agent 抛漂亮表格 |
| **JSON 入参** | 复杂参数支持 `--from-file` 和 `--json '<inline>'`，避免 Flag 爆炸 |
| **Schema 自省** | `cli schema <command>` 实时返回参数/必填/默认值/类型 |
| **Dry-Run 默认** | 写操作默认预览，必须显式 confirm 才执行 |
| **可预测错误码** | 错误码稳定、可枚举，Agent 可程序化处理 |
| **稳定动词集** | get / list / create / update / delete / wait / describe / logs |
| **分页与流式** | 大结果集强制分页，避免上下文爆炸 |
| **可断点续跑** | 长任务支持 idempotency-key，重试不产生副作用 |

举个对比例子。

旧 CLI 风格：

```bash
xxxcli create-cluster --ClusterName demo --Region ap-shanghai \
  --KubernetesVersion 1.30 --VpcId vpc-xxxx --SubnetIds subnet-xxxx,subnet-yyyy \
  --InstanceType S5.LARGE16
```

Agent-Native 风格：

```bash
# 1. 先让 Agent 拿到 Schema
cli schema cluster create

# 2. Agent 生成 JSON，一次性提交
cli cluster create --from-file cluster.yaml --json

# 3. 等待就绪
cli cluster wait --name demo --status running

# 所有输出都是 JSON，可以被 Agent 直接 parse 和决策
```

关键不是"换了语法"，而是**整个交互方式变成了 Agent 可以稳定理解的形态**。

---

## 🛠️ 第二步：用 CLI 作为骨架重写文档

如果 CLI 是稳定的、自描述的、可执行的，那文档自然应该围绕 CLI 来写。

我给团队提的重写规范是这样：

| 旧写法 | 新写法 |
|--------|--------|
| 「在控制台点击「创建集群」按钮」 | 一段可执行的 `cli cluster create --json '...'` |
| 一张控制台截图 | 一段 `cli cluster get --id <id>` 的真实输出 |
| 「选择 VPC 和子网」 | 用 JSON 参数明确写出 vpc_id 与 subnet_id |
| 「等待集群创建完成」 | `cli cluster wait --id <id> --status running` |
| 「点击「确认」即可生效」 | `cli xxx apply --confirm` |

核心约束：

> **截图只用于「概念示意」，不用于「操作步骤」。**

入门教程、操作指南、最佳实践、故障排查 —— 所有"教你怎么做"的文档，骨架都应该是 CLI。

带来的好处：

- AI Agent 可以直接抽取并执行
- 客户可以直接复制粘贴
- 文档不再受 UI 改版影响
- 同一份文档既可读，又可运行

---

## 🛠️ 第三步：给文档加上「编译」能力

这一步最关键，也是最容易被忽略的一步。

我设计了一个简单的工具链，叫 `docs-cli`（团队里实际命名可能不同）：

```bash
# 静态检查：文档里的 CLI 命令是否语法合法
docs-cli lint docs/

# 动态校验：在隔离测试环境跑一遍文档里的 CLI 序列
docs-cli verify docs/ --env testing

# 文档生成：CLI Schema → 自动生成参考文档
docs-cli generate --from-schema

# 漂移检测：文档里引用的 CLI/API 是否已废弃
docs-cli drift-check
```

然后给 Markdown 加一点元数据：

````markdown
```bash {executable=true, env=testing}
cli cluster list --output json
```
````

提交 PR 时，CI 会自动跑：

```text
产品迭代（CLI 或 API 变更）
        ↓
   文档 CI 自动触发
        ↓
   lint → 语法错？拒绝合并
   verify → 跑不通？拒绝合并
   drift-check → 引用废弃？标红警告
        ↓
   文档总能跟产品同步
```

效果：**文档不再是"写"出来的，而是"编译生成 + 自动校验"的**。

校验是分级的：

- **lint** 必过：语法错、命令不存在等
- **verify** 核心场景必过：完整教程跑通
- **drift-check** 警告级：可以延后修，但不能忽视

这是文档第一次有了类似代码的"工程化"基础设施。

## 📊 为什么这件事现在做最有意义

我把这个判断拆成三个变量：

### 1. AI Agent 工具的爆发

- Claude Code、Cursor、CodeBuddy、Gemini CLI、Codex CLI ……都在抢"AI 操作世界"的入口
- 它们都在用同一个共识：**CLI 是 AI 操作世界最稳定、最经济的接口**

### 2. 主流云厂商都开始重写 CLI

- AWS、GCP 都在做面向 Agent 的 CLI 改造
- 也有像 [OpenCLI](https://github.com/jackwener/OpenCLI) 这种社区项目，把没有 API 的网站强行 CLI 化给 Agent 调用

### 3. 文档 docs-as-code 的成熟基础

- Markdown + CI + Lint 早就是开源项目的标配
- VitePress、Docusaurus、MkDocs Material 等工具都支持自定义代码块元数据
- 把 docs-as-code 从开源项目搬到云产品文档体系，技术上已经没有门槛

三个变量叠加在一起：**现在做，能形成时间窗口优势**。等所有大厂都做了，就只是补课。

---

## 🎯 我对这件事的判断

如果做成，会带来三层价值：

### 第一层：客户接入成本下降

```text
旧链路：客户 SRE → 查文档 → 看截图 → 在控制台手动操作 → 接入 CI/CD
新链路：客户 SRE → CLI + 文档示例 → 直接 copy 到流水线 → 上线
```

接入成本会显著下降，特别是对外部客户的自动化场景。

### 第二层：AI 时代的入口卡位

```text
当 AI Agent 想"在云上创建一个集群"时：
- 如果你的 CLI 是 Agent-Native 的 → Agent 默认选你
- 如果你的 CLI 不是 → Agent 转向别家或者放弃集成
```

这件事的天花板，不是节省几个文档维护工时，而是**未来 1-2 年里，AI Agent 在云容器赛道选谁作为默认调用对象**。

### 第三层：内部协同效率

- 产品经理、TAM、SA 自己也能用 CLI 替代控制台
- 文档腐烂率显著下降，维护人力节省
- 产品迭代和文档同步从「靠人」变成「靠 CI」

## ⚠️ 我自己最担心的几个点

写策划稿过程中，反复 challenge 自己：

### 担心 1：是不是又一次「内部理想化」？

不是。这件事的关键不是「我们想做」，而是 **AI Agent 工具已经在普及，客户和市场结构已经在变化**。是先做还是被动做的问题。

### 担心 2：是不是工作量太大？

确实大，但**不需要一次性做完**。我把它拆成了 4 个阶段：

- M0：跨团队对齐 + 规范 v0.1
- M1：首批 20 个 CLI 命令 + docs-cli MVP
- M2：Top 30 篇文档 CLI 化重写 + CI 接入
- M3：标杆客户灰度 + 对外发声

第一阶段大约 4-5 个月，是一个可以验证假设、收集反馈的窗口。

### 担心 3：会不会替代控制台？

不会。控制台仍然服务人类用户，特别是探索式操作和可视化场景。

**CLI + 文档可编译化是新增的 Agent-Native 接口层，不是替换。**

### 担心 4：跨团队推不动怎么办？

这是最大的风险。这件事天然涉及 CLI 团队、文档团队、产品团队、AI 工具团队。

我的应对：

- 用「联合攻坚项」机制立项，让多个团队共同担责
- 从 Top 30 篇文档开始而不是全量重写
- 先让内部 AI 工作站接入新 CLI，跑通"产品经理自己用"的内循环
- 通过博客、技术大会等渠道做声量，让外部反向倒逼内部

## 🎉 结语

这事最终能不能成，我现在不敢打包票。

但是有一个判断我越来越确信：

> **未来云产品的"门面"，不再是控制台的首屏，而是 CLI 是否足够 Agent-Native，以及文档是否能像代码一样被编译。**

对一个产品经理来说，这是一个值得长期 push 的方向。哪怕落地有难度，把这件事提出来、想清楚、写成可执行的策划稿本身，就是一种价值。
