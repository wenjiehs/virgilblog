# 两天时间，给 Agent Memory 这块"无人区"画了一张产品地图

## 🤔 为什么突然想研究这个

最近在思考一个产品方向——如何让 Agent 真正"记住"东西。

做 Agent 产品的同事应该都有感觉：**LLM 单次调用已经非常强了，但一旦涉及跨会话、跨任务的上下文，就开始变蠢**。用户昨天说的偏好，今天再问还得重说一遍。团队上周定的编码规范，下次生成代码又忘了。

这块能力行业里叫"Agent Memory"。我知道 Mem0、Letta 这些产品的名字，但说实话，我对这个领域的认知停留在"听过几个名字"的水平。

所以我决定用两天时间，把这个领域**系统性地捋一遍**，然后尝试设计一个产品出来。

坦白说，这是一次有点"贪心"的尝试——调研+设计放在一起，对一个产品经理来说挑战不小。但我有一个信念：**产品调研如果不落到具体设计上，很容易变成"知道了很多却做不出任何东西"的陷阱**。

这篇文章记录这个过程，不是炫耀"我做了什么"，而是想分享"我怎么做的"——特别是和 AI 协作的部分，可能对做类似工作的人有参考价值。

---

## 🛠️ 第一天：4 周的调研压缩进 1 天

### 起点：一份任务清单

我没有一上来就读论文或对比产品。而是先让 AI 帮我做了一件事：**把整个领域拆解成调研任务清单**。

任务清单最后是这样的：

```
7 大调研分区：
  一、应用场景调研（4 项任务）
  二、架构与技术路线（6 项任务）
  三、云厂商 Memory 产品形态（5 项任务）
  四、创业公司 / 独立产品（5 项任务）
  五、开源社区（4 项任务）
  六、学术前沿（3 项任务）
  七、产品设计输入（5 项任务）

总计 31 项任务，原计划 4 周完成。
```

**为什么要先拆任务清单？**

因为调研最容易犯的错误是"边调研边跑偏"——读了一篇论文觉得有意思，就顺着往下读；看了一个竞品觉得牛，就开始深度拆解。结果两天过去，读了很多但没有结构。

有了任务清单，每个动作都是为了**填上清单里的某一格**，效率高很多。

### 方法：AI 不是搜索引擎，是调研助手

我的工作方式不是"让 AI 搜一下 Mem0 是什么"然后粘贴结果。而是给 AI 设定一个**调研员的角色**：

- **输入**：任务描述 + 调研维度模板
- **过程**：AI 自主搜索、阅读、提炼、交叉验证
- **输出**：结构化的综合报告 + 关键发现 + 对比表格

每周的综合报告都要求包含几个固定要素：
- 调研任务覆盖情况（哪些做了，哪些没做到）
- 关键发现（3-5 个值得记住的点）
- 对比矩阵（产品×维度的表格）
- 来源清单（每个结论背后的链接）

这样做的好处是，**报告写出来就是可以复用的资产**，不是一次性的聊天记录。

### 调研产出

两天之后，最终产出了 4 份综合报告，覆盖 31 项任务全部完成：

```
Week 1: 场景分类 + 存储架构 + 分层模型 + 5 家云厂商 + 4 家创业公司
Week 2: 检索架构 + 生命周期 + 多租户 + 开源社区 + MCP 生态
Week 3: 场景优先级 + Memory vs RAG + 评测基准 + 学术前沿 + 认知科学
Week 4: 产品定位 + API 对标 + 定价模型 + 开发者体验 + 整合方案
```

原本以为要花 4 周的事情，实际用了 2 天。这不是因为 AI 比人快 14 倍——**是因为 AI 可以并行**。我可以同时让 AI 去调研"存储架构对比"和"多租户隔离方案"，而人脑一次只能专注一个。

## 💡 第二天上午：从调研到洞察

调研的终点不是"知道了很多"，而是"提炼出几个有用的判断"。

我花了一个上午，把 4 份综合报告里的关键发现汇总，最后提炼出 **10 个对产品设计有直接影响的洞察**：

| # | 洞察 | 为什么重要 |
|---|------|-----------|
| 1 | **工程记忆是全行业蓝海** | 所有 Memory 产品都聚焦对话上下文，代码/CI/部署的工程信号无人覆盖 |
| 2 | **可观测性是开发者体验的新焦点** | "调试记忆比调用 API 难 10 倍"——这是行业空白 |
| 3 | **整合被严重忽视** | "存储一切"反而会让检索变差 |
| 4 | **Memory 不等于 RAG** | 很多团队混用，但这是根本性的误解 |
| 5 | **"弃权能力"评估空白** | Agent 识别"自己不知道"很重要，但没人专门做 |
| 6 | **三信号加权是基础范式** | 相关性+时效性+重要性，这是共识 |
| 7 | **双层架构是生产共识** | 热路径+冷路径+后台整合 |
| 8 | **选择性记忆成主流** | 牺牲一点准确率换 90% 成本降低 |
| 9 | **艾宾浩斯遗忘曲线已工程化** | 有可以直接用的公式 |
| 10 | **场景优先级清晰** | Coding Agent 是最合适的切入点 |

这 10 个洞察就是后面产品设计的"指南针"。

**我最喜欢第 1 条**：所有人都在卷"对话记忆"，但代码、提交、部署、故障这些工程信号没有人专门做。对一个做开发者工具的团队来说，这是一个天然的切入点。

## 🎯 第二天下午：产品定义讨论

有了 10 个洞察之后，我开始和 AI 做产品定义。

这个过程不是我让 AI 写一份文档那么简单。而是**一轮又一轮的对话**——AI 提出选项，我做决策，AI 追问假设，我澄清思路。最后沉淀了 **8 项核心产品决策**：

```
D1 集成方式    → 可选模块（不强制所有 Agent 启用）
D2 分类主导    → 记忆类型优先
D3 分类维度    → 内容性质 × 生命周期（正交）+ 作用域（属性）
D4 三种记忆    → 事实 / 事件 / 程序（对齐认知科学）
D5 用户感知    → 分类不暴露，体现为"Agent 更懂业务"
D6 治理融合    → 能力+治理密不可分
D7 交互界面    → 混合三层（API + 声明式 + 控制台）
D8 构建者模型  → 混合模式（人定义体系，Agent 执行操作）
```

最核心的是 **D8**，也是讨论最久的一项。这个问题本质上是哲学级的：**谁来构建 Agent 的 Memory 体系？**

三种极端选项：
- **系统全自动**（Mem0 风格）：开发者失去控制权
- **Agent 全自治**（Letta 风格）：Agent 可能记乱，无法对齐业务
- **混合模式**：人定义"记什么、怎么记"，Agent 执行"何时记、记什么内容"

我选了第三个。理由是：**"记什么"本质是业务决策，不是 AI 决策——由人定义才对**。Agent 自主构建时，本质上是在模拟人的构建逻辑，所以人的蓝图才是源头。

这个决策贯穿了后面所有接口设计。

### 核心叙事

8 项决策背后有一个核心叙事，花了最久才提炼出来：

> **不是卖"智能"，是卖"可控的智能"。**

ToC 场景追求"魔法体验"——用户不感知记忆怎么运作。但 ToB 场景完全不一样——企业客户关心的是：

- 记了什么？（可见）
- 谁能看？（权限）
- 能删吗？（合规）
- 出错怎么办？（治理）

这些需求不是附加功能，而是**第一天就必须在产品里**的。这也是 D6 "能力和治理融合设计"的出发点。

## 🎛️ 第二天晚上：接口设计

产品定义是"What"，接口设计是"How"。

到这一步，我其实有点累了。但我想趁热打铁把接口也定下来——否则明天再回来就得重新加载上下文。

这次对话我主动引入了**兼容层**的概念。我告诉 AI：

> 我同意混合模式，但我不想让客户必须学一套新 API。未来应该有一层转译，让用 Mem0、Letta 写的代码能直接跑在我们的系统上。

这个决策后来被证明很重要。AI 追问了我几个问题：

- **兼容范围**：API 语法兼容？还是行为完全等价？
- **兼容位置**：独立 SDK 还是网关级转译？
- **兼容优先级**：先兼容谁？

我选的是"智能适配 + 独立 SDK + 先兼容 Mem0 和 MCP"。

### 接管下游

这里我想强调一下"智能适配"这个选择的战略价值。

Mem0 有 48K stars，Letta 有 21K stars。传统思路是"做一个比他们更好的 API 来竞争"。但我选的是另一条路：

> **不和 Mem0 竞争 API 设计权，而是接管他们的下游。**

具体就是做一个 `ags-memory-mem0-compat` 的包，让用户只需要 `pip uninstall mem0ai && pip install ags-memory-mem0-compat`，**一行代码都不用改**。

代码不变，但用户获得了：
- 审计日志（原本没有）
- 多租户隔离（原本没有）
- 矛盾检测（原本没有）
- Token 节省数字（原本看不到）

一周后用户主动去看控制台，然后问："怎么解锁更多能力？" → 产品拉新闭环就闭上了。

这是典型的"**以斗争求合作**"策略——不正面对抗，但通过兼容把对方的生态变成自己的用户基础。

### 五层接口

最后设计出来的接口是五层：

```
设计时 (Design Time)
  ┌───────────────────────────────────────────────────┐
  │ 界面 A: 声明式配置 (memory.yaml)                    │
  │   "我的 Agent 需要什么类型的记忆？怎么治理？"         │
  └───────────────────────────────────────────────────┘

运行时 (Runtime)
  ┌──────────────────────┐  ┌──────────────────────┐
  │ 界面 B: 开发者 SDK    │  │ 界面 C: Agent 工具集  │
  │ (Python/TS)          │  │ (LLM tool calling)   │
  │  应用层调用           │  │  Agent 自主读写       │
  └──────────────────────┘  └──────────────────────┘

治理时 (Governance Time)
  ┌──────────────────────┐  ┌──────────────────────┐
  │ 界面 D: 管理控制台    │  │ 界面 E: 可观测仪表盘  │
  │  策略/审计/权限       │  │  质量/成本/告警       │
  └──────────────────────┘  └──────────────────────┘

兼容层 (Compatibility Layer)
  ┌───────────────────────────────────────────────────┐
  │ 界面 F: 兼容 SDK (Mem0/MCP 零迁移成本)             │
  └───────────────────────────────────────────────────┘
```

每层接口都详细设计了——YAML Schema、Python SDK 方法签名、Agent 可调用的工具集（`recall / observe / reflect / propose_forget / suggest_schema`）、控制台的审批队列 UI、仪表盘的指标……

最后文档写了 1000+ 行。

## 🧪 用 Coding Agent 场景走一遍

设计完之后最怕的是"自洽但不可用"。所以最后我又花了点时间，**用一个真实场景把接口走一遍**。

我选的是 Coding Agent 场景。模拟了 4 个真实故事：

**故事 1：新员工上手**

```
新员工说："帮我写一个 user 服务的 API"

Agent 内部：
  1. 调 memory_recall 查团队约定 → 返回 RESTful 风格、GORM、错误码规范
  2. 基于约定生成代码
  3. 调 memory_observe 记录这次生成

整个过程对新员工透明，但团队约定被自动应用了。
```

**故事 2：Agent 发现重复模式**

```
Agent 在一周内观察到 10 次部署事件，发现：
  "团队习惯周一~周五 9AM 部署"

系统推断：这是一个团队约定（程序记忆）。
但因为配置了 require_approval: true，
不直接入库，而是发起审批请求。

Tech Lead 在控制台批准 → 新记忆生效。
```

**故事 3：Agent 发现过时记忆**

```
旧记忆："User uses Python 2.7"
新观察：过去 30 天用户全是 Python 3.11+

Agent 主动调 memory_propose_forget：
  "这条记忆可能过时了，建议删除。证据：..."

Tech Lead 审批 → 删除。
```

**故事 4：外部 Mem0 用户无感迁移**

```
一个团队原本用 Mem0 写的 Agent：
  pip uninstall mem0ai
  pip install ags-memory-mem0-compat
  # 代码一行都没改

一周后他们打开控制台，发现：
  审计日志✅ 多租户隔离✅ 矛盾检测✅ Token 节省数字✅
  → "怎么解锁更多能力？"
```

**走完这 4 个故事，我确认接口设计自洽了。**

特别是前 3 个故事，完美体现了 D8 混合模式——Agent 有"建议权"和"申请权"，但"决定权"始终在人手上。这就是**"可控的智能"**的具象化。

## 💭 两天之后的真实感受

### 关于 AI 协作

这两天最深的感受：**AI 不是工具，是一个高度能干但需要方向的同事**。

我的角色变成了：
- **定义问题**：要解决什么、边界在哪里
- **设定方法**：用什么框架、产出什么格式
- **做关键判断**：选项之间怎么选、trade-off 怎么权衡
- **验证产出**：逻辑自洽吗、和调研数据对齐吗

AI 负责：
- **执行具体任务**：搜索、阅读、归纳、对比
- **提供选项**：每个决策点列出可能的选择
- **追问假设**：发现我思路里的模糊地带
- **沉淀文档**：把结论写成结构化文档

两者结合才是高产出的。

### 关于调研的陷阱

**调研最大的陷阱是"自我满足"**——觉得读了很多东西就是有产出。但其实调研的唯一价值是**支撑决策**。

如果调研结束后无法回答 "所以我们应该怎么做？"，那这次调研就是失败的。

我这两天强迫自己在每周综合报告里都写"对 AGS 产品设计的启示"这一节。就是为了防止调研变成"知识收集"。

### 关于"写得太快"的隐忧

**我知道这份设计文档可能过于乐观。**

两天时间做出来的接口设计，不可能在所有场景都经得起考验。它只是一个 v0.1 起点。真正的考验在后面：

- 拉技术团队 review，会发现工程实现的约束
- 找潜在客户访谈，会发现使用场景的盲点
- 和商业团队对齐，会发现定价和 GTM 的挑战

**文档的速度只决定起点，不决定终点**。

但我也不焦虑——因为有了这个起点，后面每一次迭代都有基础。比"白纸一张"好太多。

## 🚀 接下来做什么

这两天做的事情，本质上是完成了 **What（产品定义）+ How（接口设计）** 的闭环。接下来有几个方向：

4. **技术团队评审**：拿着文档找有经验的工程师吐槽

## 🎉 总结

两天时间，给 Agent Memory 这块领域画了一张产品地图：

- 4 份综合调研报告，覆盖 31 项任务
- 10 个关键洞察
- 8 项核心产品决策
- 1 份完整的接口设计稿
- 1 个走通的场景验证

最大的收获不是这些产出本身，而是验证了一种工作方式：**AI 辅助下的"压缩式调研+决策驱动设计"**。

以前做一个类似深度的产品分析可能需要 2-4 周，现在 2 天就能出 v0.1。这个速度差异不是量的变化，是质的变化——意味着产品经理可以**在同一个季度里探索多个方向**，而不是押宝在一个方向上。

当然，这也意味着对产品经理的要求更高了：
- 以前慢慢来也能做好，现在必须高密度做决策
- 以前 AI 不行必须自己做，现在必须判断 AI 什么时候靠得住
- 以前写文档就够了，现在要写可以 ship 的决策链

**工具变了，工作方式变了，但对判断力的要求不降反升。**
