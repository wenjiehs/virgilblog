# 给 Codex 装上个人规划 OS：一次把习惯做成插件的实践

这次做插件，起点其实不是技术问题，而是一个很普通、也很容易反复失败的个人管理问题：

> 我是不是应该养成制定计划、梳理每周 todo、定期 review 的习惯？

这个问题听起来很像自我管理鸡汤，但落到我自己的工作方式里，它其实挺具体。

我平时会同时在多个项目、多个仓库、多个产品方向之间切换。很多时候不是没有计划，而是计划被散落在对话里、项目文档里、临时笔记里。等到下一周再回头看，已经很难判断：这一周到底有没有沿着主线推进？哪些事情只是看起来紧急？哪些项目其实应该放进机会池，而不是进入本周执行清单？

所以这次我没有先去找一个现成的 todo 工具，而是尝试把这件事做成一个 Codex 插件：让规划、更新、复盘和可视化，变成一个跨项目都能唤起的工作流。

## 🤔 为什么要插件化

一开始我其实也想过：是不是建一个 Markdown 目录就够了？比如放一个 `year-plan.md`，再建 `weekly/`、`monthly/`、`projects.md`。

但很快就发现，只靠目录不够。

因为我真正需要的不是一个“保存文件的地方”，而是一套会在不同项目里持续提醒我的工作方式：

- 在任何项目里都能问：“这件事和今年主线一致吗？”
- 每周能生成本周主线、关键成果和明确不做事项。
- 做 review 时不只是总结，而是判断有没有跑偏。
- 新增项目时，不藏在当前 repo，而是进入统一项目池。
- 修改规划后，dashboard 自动更新，不要靠我手动记得刷新。

这些动作明显不是某个业务项目的一部分。它更像一个个人操作系统，应该独立于项目存在，又能在所有项目里被调用。

所以最后定下来的形态是：

```text
Codex 任意项目
  -> 唤起 personal-planning-os
  -> 读取个人规划库
  -> 更新年度/月度/周度/项目/决策文件
  -> 自动生成 dashboard.html
```

这个设计的核心不是“做一个插件很好玩”，而是把一个容易中断的习惯，变成 Codex 的默认工作流。

## 🎯 先写验收标准，而不是先写插件

这次比较有意思的一点是，我没有让 Codex 直接开始写文件，而是先让它制定验收标准。

最后的验收标准大概包括这些：

1. 插件名固定为 `personal-planning-os`。
2. 能在任意项目中触发年度规划、周计划、月度复盘、项目更新和主线校准。
3. 有一个独立的规划库，作为长期 source of truth。
4. 规划库至少包含：年度规划、项目池、决策日志、weekly、monthly、inbox。
5. review 必须产出判断和下一步动作，不能只是总结。
6. todo 必须写成下一步行动，不能写成愿望清单。
7. 修改任何规划 Markdown 后，必须同步更新 dashboard。
8. 插件结构必须通过校验，并重新安装到 Codex。

这个步骤看起来像流程，但它其实很关键。

如果不先定义验收标准，很容易做成一个“看起来有用”的插件：能写几段规划、能生成几个文件，但真正使用时还是靠人记得维护，最后又变回一个冷掉的文件夹。

我想要的是：以后我只要说“帮我 review 这周有没有跑偏”，Codex 就知道该读什么、改什么、生成什么、最后给我什么判断。

## 🛠️ 实际做出来的结构

插件最后做得很轻，没有上复杂服务，也没有接数据库。

### 1. 插件本体

插件本体里主要有三类内容：

```text
personal-planning-os/
  .codex-plugin/plugin.json
  skills/personal-planning-os/SKILL.md
  scripts/generate_dashboard.py
```

`plugin.json` 负责让 Codex 识别这是一个插件；`SKILL.md` 负责定义什么时候触发、怎么工作；`generate_dashboard.py` 负责把 Markdown 规划库生成一个本地 HTML dashboard。

我比较喜欢这个边界：

- **Skill 管工作流**：什么时候读文件，什么时候更新，什么时候复盘。
- **Markdown 管长期内容**：规划、项目、决策都保存在可读文件里。
- **脚本管确定性输出**：dashboard 不靠每次临场发挥，而是用脚本稳定生成。

### 2. 规划库

规划库的结构也很朴素：

```text
personal-planning/
  year-plan.md
  projects.md
  decision-log.md
  weekly/
  monthly/
  inbox/
  dashboard.html
```

这里最重要的是 `weekly/` 和 `decision-log.md`。

`weekly/` 是执行入口。年度规划写得再好，如果不能落到“本周主线是什么、哪三件事完成就算没白过、哪些事明确不做”，它很快就会变成装饰。

`decision-log.md` 则是为了防止反复纠结。很多时候真正消耗注意力的，不是做事本身，而是同一个选择反复拿出来想。把“为什么开始”“为什么暂停”“为什么放弃”记下来，其实是在给未来的自己减负。

### 3. dashboard

第一版 dashboard 是一个单文件 HTML，不需要服务，直接打开就能看。

它展示几块内容：

- 年度主线
- 当前周主线和待办状态
- 项目池
- 项目矩阵
- 本周三件关键成果
- 不做事项
- 风险和跑偏信号
- 约束和反目标
- 决策日志
- 月度复盘时间线

这不是为了做一个漂亮报表，而是为了让自己一眼判断：现在的注意力是不是还在主线上。

## 🔁 一个小转折：dashboard 不能手动更新

做到 dashboard 之后，我又意识到一个问题：如果每次改完规划还要手动生成 dashboard，那这件事很快就会被忘掉。

所以后面把规则写进了 skill：

> 只要本 skill 创建或修改任何规划 Markdown 文件，必须在同一轮回复前重新生成 dashboard。

这条规则看起来很小，但它让 dashboard 从“附加产物”变成了“工作流的一部分”。

否则 dashboard 很可能会变成另一种坏掉的仪表盘：样子还在，但数据已经过期。

## 🌏 另一个小坑：默认语言

第一版做完以后，dashboard 里很多内容是英文：Annual main line、Todo state、Project Pool、Decision Log……

这其实很别扭。

因为这是我的个人规划系统，不是英文技术文档。每周打开它的人是我自己。如果标题和默认文案都是英文，就会多一层认知切换，也不符合我平时复盘的语言习惯。

所以后面又做了一次中文化：

- 插件描述改成中文。
- skill 规则明确：默认用中文保存规划内容和生成 dashboard。
- 模板章节改成中文，比如“年度主线”“本周主线”“不做事项”“风险和跑偏信号”。
- 生成器兼容旧英文标题，但新内容默认中文。
- 已有规划库也转成中文章节和中文内容。

这个改动让我有个很实际的感受：工具不是越通用越好。个人工作流工具首先要贴合自己的语言和习惯。否则它越“标准”，越不像自己的东西。

## ✅ 验证：不只是生成文件

这次过程中，我让 Codex 做了几类验证。

第一是插件结构校验。插件和 skill 都跑了校验脚本，确保 `plugin.json`、frontmatter 和目录结构能被 Codex 正常识别。

第二是重新安装。每次修改插件后，都通过 cachebuster 重新安装，让新线程能拿到最新版本，而不是只改了本地源文件。

第三是用安装缓存里的生成器重新跑 dashboard。这个验证很重要，因为它证明真正被 Codex 安装的那份插件，也能生成 dashboard。

第四是截图预览。桌面版 dashboard 用无头 Chrome 截过图，确认不是空白页面，也不是旧版英文界面。

移动端这里有个小插曲：长英文产品名在窄屏截图里容易撑出裁切。后来做了一些换行和卡片化处理，但我最后没有为了移动端把桌面版 dashboard 改得很丑。对我来说，这个 dashboard 第一使用场景还是桌面。如果未来真要手机看，应该单独做一个“窄屏摘要版”，而不是硬把所有信息塞进小屏。

## 💭 这次让我重新理解了“插件”

以前我更容易把插件理解成“给 AI 增加一个能力”。

但这次更像是：**给自己增加一条可重复的工作路径**。

插件真正有价值的地方不是它能不能写 Markdown，而是它把一些容易被遗忘的动作固定下来：

- 做规划前先读年度主线。
- 周计划必须写“不做事项”。
- review 必须给出是否跑偏的判断。
- 项目变化要进入统一项目池。
- 重要选择要写入决策日志。
- 修改规划后 dashboard 必须同步。

这些动作单独看都很小，但组合起来，就从“偶尔整理一下计划”变成了一套闭环。

## 🤔 给自己的几个提醒

第一，不要把规划系统做得太重。

我以前很容易被工具诱惑：想加数据库、加图表、加提醒、加自动分析。但这次更应该先守住最小闭环：周计划、周复盘、月度复盘、决策日志、dashboard。

第二，todo 不要写成愿望清单。

“推进副业”“研究方向”“整理资料”都太虚。真正有用的 todo 应该是下一步动作，比如“整理 3 个可联系的需求来源”“把某个项目加入候选池并写清复盘日期”。

第三，复盘不是为了写好看的总结。

如果一篇 review 不能帮助我砍掉事情、调整下周主线、更新项目优先级，那它只是情绪记录。情绪记录也有价值，但不是这套系统的核心。

第四，中文默认很重要。

这是一个小细节，但长期使用时会很明显。个人系统一定要用自己最自然的语言，否则会产生额外阻力。

## 📊 实际产出

这次实际沉淀了几样东西：

| 产物 | 作用 |
| --- | --- |
| Codex 插件 | 在任意项目中唤起规划和复盘工作流 |
| 个人规划库 | 保存年度、月度、周度、项目和决策内容 |
| dashboard 生成器 | 从 Markdown 自动生成可视化驾驶舱 |
| 中文模板 | 保证后续保存和展示默认中文 |
| 验收标准 | 约束插件不是“能用一次”，而是能持续使用 |

## 🎉 小结

这次实践对我最大的启发是：习惯不一定只靠意志力养成，也可以被做成工具链。

如果一个动作每周都要做，而且每次都容易忘掉关键步骤，那它就很适合沉淀成插件、脚本或者模板。

这次的 `personal-planning-os` 还很轻，甚至有点朴素。但我反而觉得这是它的优点：Markdown 可读、HTML 可打开、脚本可重跑、插件可跨项目触发。

它不是一个完美的个人管理系统，更像是一个开始形成节奏的脚手架。

对我来说，下一步不是继续加功能，而是真正用它跑几周：每周一制定主线，周中校准，周五复盘，月末再看一次方向有没有偏。

如果这个节奏能跑起来，插件才算真的有价值。
