# 给 TCR 产品策划工作站做了一次大升级：从 3 个场景扩展到 6 个

## 为什么要动这个工作站

工作站最初的设计很简单：三个场景，三个 Agent。

- 产品逻辑归档（knowledge-manager）
- 功能策划（product-architect）
- 客户损益分析（data-analyst）

用了一段时间，发现一个问题——我的日常工作远不止这些。

比如写 PR 稿。每次跟 AI 一起写公众号文章，我都得重新解释一遍"TCR 是什么、这个功能的背景是什么"，很烦。工作站明明已经有十几篇产品逻辑文档了，但 AI 不知道主动去查。

再比如分析机构问卷。IDC、Gartner 这些机构定期发来评估问卷，问题都是他们自己的语境，比如"Deployment Gating"、"Geo-replication"，但我们的产品术语是"部署阻断"、"多地域复制"。每次答题都要手动做这个翻译工作，费时费力。

还有客户需求管理。客户反馈散落在工单、销售反馈、访谈记录里，没有结构化，哪个需求该优先做，凭记忆和感觉，不靠谱。

加上之前一直在往知识库里加内容，文档越来越多，Agent 找资料越来越慢，也需要一个更好的索引机制。

问题列出来，就知道今天该做什么了。

---

## 做了哪些事

### 1. 建全局索引（最基础，最重要）

第一件事不是加新功能，而是先把"导航"做好。

以前知识库有一个 `docs/05-knowledge-base/index.md`，但那个索引是人工维护的，Agent 用起来不顺手——它不知道"写 PR 稿应该先读哪几个文件"，也不知道"答问卷要去哪里找安全能力的描述"。

所以专门建了一个 `docs/00-index/global-index.md`，定位是**所有 Agent 的第一个查阅点**。

这个索引包含：
- 9 个知识目录的作用速查表
- 每篇文档的关键词（方便模糊定位）
- 按任务类型的知识调用路径（写 PR 稿要读哪几个文件、答问卷要读哪几个文件）
- 文档命名规范速查

以后每个 Agent 的工作流第一步统一是：查全局索引，定位文件，再读内容。

---

### 2. 对外内容创作统一管理

之前写公众号文章有规则，但没有 Agent 统一负责，PR 稿、发版公告、官网文案的目录也没规划好。

这次新建了 `content-creator` Agent，统一接管所有对外内容。

工作流大概是这样：

```
接收任务
  → 查全局索引（定位相关产品知识）
  → 读历史稿件（保持表述一致）
  → 按对应模板创作
  → 质量审核（禁止内部链接、竞品数据等）
  → 归档到对应目录
```

对应的三个目录和三个模板也一起建好了：
- `docs/06-marketing/pr/` — PR 稿和公众号文章
- `docs/06-marketing/release-notes/` — 发版公告
- `docs/06-marketing/official-site/` — 官网文案

实际用起来的感受是：解放感挺强的。以前每次写 PR 稿都要先跟 AI 解释产品背景，现在直接说"帮我写一篇关于镜像签名功能的公众号文章"就行了，背景知识 Agent 自己去查。

---

### 3. 客户需求管理

这是一个纯粹的效率问题，解法也比较直接。

新建了 `requirements-manager` Agent，触发方式就是描述客户反馈：

```
"客户反馈：某金融行业客户希望 TCR 支持多个实例之间的批量镜像同步，
目前只能逐个配置，管理成本很高..."
```

Agent 会自动生成一张结构化的需求卡片，包含：
- 客户原始反馈（原话保留）
- 产品视角的需求理解
- 五维优先级评分（客户价值、覆盖范围、商业价值、实现难度、战略契合）
- 处理决策和跟踪记录

卡片按状态分入三个目录：`incoming/`（待评估）、`backlog/`（已纳入规划）、`archived/`（已完成或已拒绝）。

有一个细节处理得比较细：客户信息脱敏。规则里规定用行业类型替代公司名（"金融行业某大型企业"而不是具体公司名），不记录联系人信息，这样需求卡片可以相对开放地在团队内流通。

---

### 4. 分析机构问卷答题

这是我觉得今天最有价值的改动，解决的是一个实际痛点。

IDC、Gartner 的问卷特点是：问题用他们自己的框架和术语，但我们需要用产品的实际能力来回答。这个"翻译"工作之前全靠人工，每次都要重新做一遍。

新建的 `survey-responder` Agent 核心做了两件事：

**第一，建了术语对照表。** 把分析机构常用术语和 TCR 产品术语一一对应，比如：

| 分析机构术语 | TCR 对应 |
|------------|---------|
| Deployment Gating | 高危镜像部署阻断 |
| Geo-replication | 同实例多地域复制 |
| Supply Chain Security | 软件供应链安全 |
| Lazy Loading | 镜像按需加载（Nydus） |

以后答题时直接查这张表，不用每次现想。

**第二，定了能力状态表述规范。** 这点很重要——只描述已正式发布的功能，规划中的用"在产品路线图中规划..."表述，绝对不夸大。分析机构如果发现答题内容和实际产品不符，会直接影响评级结果，这个坑不能踩。

## 整理下来的感受

**工作站的价值在于固化经验。**

比如问卷答题的术语对照表，是根据以往答题经验总结的。现在把它写进 Agent 的规则里，以后每次都能用到，不会因为换人或者时间久了就遗忘。工作站积累的不只是文档，也是做事的方法。

**全局索引是基础设施，值得专门投入。**

这是今天最不起眼、但我认为价值最大的改动。知识库再完整，Agent 找不到也没用。有了全局索引，每个新 Agent 的工作流起点是清晰的，不需要每次都让它自己摸索。

**Agent 之间的协作比单个 Agent 更重要。**

比如 `content-creator` 写 PR 稿，它会先查全局索引，再读产品知识文档，最后参考历史稿件保持一致性——这是三个模块协同完成的。工作站的价值不在于某一个 Agent 有多强，而在于整体能不能顺畅地跑起来。

## 还差什么

有两件事需要我自己参与才能完成，暂时没法自动化：

**设计原则和产品哲学归档。** TCR 的设计理念、产品哲学，需要我自己口述出来，再由 knowledge-manager 归档到知识库。这部分我还没做，但想做好是因为这些内容在写 PR 稿、回答问卷时能直接用上。

**客户损益分析。** 模板和 Agent 都就绪了，等有真实客户数据的时候触发。
