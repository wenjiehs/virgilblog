# 给两个 AI 产品策划工作站做了一次大升级：从 3 个场景扩展到 6 个

## 背景

我在用 AI 维护两个产品策划工作站，一个是 TCR（腾讯云镜像仓库），一个是 TKE（腾讯云容器服务）。

两个工作站最初的设计都很简单：三个场景，三个 Agent。

- 产品逻辑归档（knowledge-manager）
- 功能策划（product-architect）
- 客户损益分析（data-analyst）

用了一段时间，发现同一个问题——我的日常工作远不止这些。

两个工作站遇到了几乎相同的瓶颈：

**写 PR 稿**。每次跟 AI 一起写公众号文章，我都得重新解释一遍产品背景，工作站明明已经有十几篇产品逻辑文档了，但 AI 不知道主动去查。TCR 如此，TKE 也如此。

**分析机构问卷**。IDC、Gartner 这些机构定期发来评估问卷，问题都是他们自己的语境——TCR 那边有"Deployment Gating"、"Geo-replication"，TKE 那边有"Cluster Autoscaling"、"Node Lifecycle Management"……每次答题都要手动做术语翻译，费时费力。

**客户需求管理**。客户反馈散落在工单、销售反馈、访谈记录里，没有结构化，哪个需求该优先做，凭记忆和感觉，不靠谱。两个产品都有这个问题。

问题一样，解法也就大差不差。今天同时给两个工作站做了扩展。

---

## 做了哪些事

### 1. 对外内容创作统一管理

之前写公众号文章有规则，但没有 Agent 统一负责，PR 稿、发版公告、官网文案的目录也没规划好。

两个工作站都新增了内容创作 Agent（TCR 叫 `content-creator`，TKE 叫 `pr-writer`），统一接管所有对外内容。

工作流大概是这样：

```
接收任务
  → 查全局索引（定位相关产品知识）
  → 读历史稿件（保持表述一致）
  → 按对应模板创作
  → 质量审核（禁止内部链接、竞品数据等）
  → 归档到对应目录
```

两个工作站各自建好了对应的内容目录和模板：

- TCR：`docs/06-marketing/pr/`、`release-notes/`、`official-site/`
- TKE：`docs/07-pr-content/release-notes/`、`customer-stories/`、`tech-articles/`、`blog/`

实际用起来的感受是：解放感挺强的。以前每次写 PR 稿都要先跟 AI 解释产品背景，现在直接说需求就行了，背景知识 Agent 自己去查。

---

### 2. 客户需求管理

这是一个纯粹的效率问题，解法也比较直接。

两个工作站都新建了 `requirements-manager` Agent，触发方式就是描述客户反馈：

```
TCR 示例：
"客户反馈：某金融行业客户希望 TCR 支持多个实例之间的批量镜像同步，
目前只能逐个配置，管理成本很高..."

TKE 示例：
"客户反馈：某客户有 100+ 节点，每次手动逐个升级节点很麻烦，
希望支持批量节点升级..."
```

Agent 会自动生成结构化的需求卡片，包含：
- 客户原始反馈（原话保留）
- 产品视角的需求理解
- 优先级评估（业务影响、客户覆盖范围、紧迫程度三个维度）
- 处理决策和跟踪记录

需求按优先级归档：TCR 按状态（incoming/backlog/archived），TKE 按优先级（P0/P1/P2/P3）。

有一个细节：客户信息脱敏。规则里规定用行业类型替代公司名，不记录联系人信息，这样需求卡片可以相对开放地在团队内流通。

P0/P1 的需求还可以一键联动 `product-architect` Agent，直接生成功能策划稿，省掉中间的人工传递。

---

### 3. 分析机构问卷答题

这是我觉得今天最有价值的改动，解决的是一个实际痛点。

两个工作站都新建了问卷答题 Agent（TCR 叫 `survey-responder`，TKE 叫 `analyst-qa`）。

核心做了两件事：

**第一，建术语对照表。** 把分析机构常用术语和自己产品术语一一对应。举几个例子：

TCR 的术语对照：

| 分析机构术语 | TCR 对应 |
|------------|---------|
| Deployment Gating | 高危镜像部署阻断 |
| Geo-replication | 同实例多地域复制 |
| Supply Chain Security | 软件供应链安全 |
| Lazy Loading | 镜像按需加载（Nydus） |

TKE 的类似逻辑也一样——Gartner 问"是否支持 AI workload auto-scaling"，Agent 知道去找 TKE 的 GPU 弹性伸缩文档，而不是用通用语言含糊回答。

**第二，定能力状态表述规范。** 只描述已正式发布的功能，规划中的统一用"在产品路线图中规划..."表述，绝对不夸大。分析机构如果发现答题内容和实际产品不符，会直接影响评级，这个坑不能踩。

答题记录也会归档，方便下次相似问题保持口径一致。

---

### 4. 全局索引（TCR 工作站的额外工作）

TCR 工作站额外做了一件事：建全局索引 `docs/00-index/global-index.md`，定位是**所有 Agent 的第一个查阅点**。

这个索引包含：
- 各知识目录的作用速查表
- 每篇文档的关键词（方便模糊定位）
- 按任务类型的知识调用路径（写 PR 稿要读哪几个文件、答问卷要读哪几个文件）

以后每个 Agent 的工作流第一步统一是：查全局索引，定位文件，再读内容。这对 TKE 工作站同样适用，后续也会补上。

## 整理下来的感受

**两个工作站同时升级，反而更清晰。**

单独升级一个的时候容易陷进具体细节。两个一起做，更容易看到"同类问题的通用解法"——比如客户需求管理，TCR 和 TKE 遇到的是一样的问题，解法也是同一套思路的变体，差异只在具体的术语和目录结构上。

**工作站的价值在于固化经验。**

问卷答题的术语对照表，是根据以往答题经验总结的。现在把它写进 Agent 的规则里，以后每次都能用到，不会因为换人或者时间久了就遗忘。工作站积累的不只是文档，也是做事的方法。

**Agent 之间的协作比单个 Agent 更重要。**

比如客户需求管理 Agent 接收到 P1 需求后，可以直接唤起产品架构师 Agent 生成策划稿——这是两个 Agent 协同完成的。工作站的价值不在于某一个 Agent 有多强，而在于整体能不能顺畅地跑起来。

## 还差什么

**竞品知识库。** 两个工作站目前都只懂自己，不懂竞品。做战略规划、回答问卷的时候，得靠自己凭印象说"我们和 EKS / ECR 的差距在哪"，很不可靠。下一步打算给两个工作站都加竞品雷达——结构化的竞品档案 + 定期自动更新 + 竞品分析 Agent。

**产品设计原则归档。** TCR 和 TKE 的设计理念、产品哲学，需要自己口述出来，再由 knowledge-manager 归档。这部分还没做，但在写 PR 稿、回答问卷时能直接用上。

**客户损益分析。** 模板和 Agent 都就绪了，等有真实客户数据的时候触发。
