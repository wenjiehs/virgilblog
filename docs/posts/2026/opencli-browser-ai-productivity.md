# 给浏览器装上 CLI：我用 OpenCLI 做了一次产品经理日常提效实验

## 🤔 为什么突然折腾这个

产品经理的很多工作，并不是坐在那里想战略、画路线图。

更多时候，是在浏览器里反复切换各种系统：

- 查某个客户是否已经开通能力
- 看某个灰度功能还有多少账号在用
- 给某个白名单加一个测试账号
- 查商品状态、折扣配置、价格策略
- 把查到的信息再复制到文档、需求单、群聊里

这些动作单个看都不复杂，但一天做下来很消耗注意力。尤其是内部管理后台，经常没有公开 API，登录态又依赖浏览器，想自动化也不容易。

最近看到 [OpenCLI](https://github.com/jackwener/OpenCLI) 这个项目，定位很直接：

> Make Any Website & Tool Your CLI.

简单说，就是把网站、桌面应用、本地工具变成命令行，让 AI Agent 可以稳定调用。

这句话对我很有吸引力。因为我一直在做产品策划工作站、Agent Skill、自动化工作流这些事情，但之前更多是围绕"文档"和"代码仓库"展开。浏览器里的那些后台操作，仍然是一个很大的盲区。

于是我花了一个晚上做了个小实验：**能不能把一个内部白名单管理系统，做成 OpenCLI 适配器，让 AI 直接查询和操作？**

这篇文章记录这次实验。文中涉及的内部系统名称、链接、账号和具体数据都做了脱敏处理。

---

## 🎯 我想解决什么问题

我选的第一个场景是"白名单管理"。

这是一个很典型的产品经理日常操作：

1. 按业务名称、KEY 或账号搜索白名单策略
2. 查看我负责的白名单列表
3. 查看某个白名单下已经添加了哪些账号
4. 查看白名单策略详情，比如负责人、状态、过期时间
5. 必要时向某个白名单里添加一个账号

过去这些操作基本都是：打开浏览器 → 登录系统 → 找入口 → 搜索 → 点详情 → 复制信息 → 回到聊天窗口或文档。

我希望变成这样：

```bash
# 查我负责的白名单
opencli internal-whitelist mine

# 搜某个功能相关的白名单
opencli internal-whitelist search --key "feature-keyword"

# 看某个白名单下的账号列表
opencli internal-whitelist accounts --id 1234

# 给某个白名单添加账号（默认 dry-run，确认后才写入）
opencli internal-whitelist add-account --id 1234 --account <account-id> --confirm
```

从产品经理视角看，这个目标不是"炫技"，而是想验证一个更朴素的问题：

> **如果浏览器后台也能变成 CLI，AI Agent 能不能真的帮我处理一部分运营型工作？**

## 🛠️ 实际动手过程

### 第一步：安装 OpenCLI 和浏览器桥接

OpenCLI 的安装不复杂，真正关键的是浏览器桥接。

它的工作方式大概是：

```text
CodeBuddy / 终端
  ↓
OpenCLI 命令
  ↓
本地 daemon
  ↓
Chrome Extension
  ↓
已登录的浏览器页面
  ↓
目标网站
```

我本机已经有比较新的 Node.js，所以直接安装：

```bash
npm install -g @jackwener/opencli
```

安装后跑了一下诊断：

```bash
opencli doctor
```

一开始核心 CLI 是好的，但浏览器扩展没有连上。装好 Chrome 扩展后，再跑 `doctor` 就全绿了。

为了安全起见，我没有直接用主力浏览器，而是单独建了一个 Chrome Profile 专门给 OpenCLI 用。这样做有两个好处：

- 登录态和日常浏览器隔离
- 以后如果要停用，直接删这个 Profile 就行

这个细节挺重要。因为 OpenCLI 的优势就是复用浏览器登录态，但这也意味着它和账号安全、内部数据安全紧密相关。

### 第二步：先跑一个公网命令确认链路

在碰内部系统之前，我先用一个无登录的公网命令确认 OpenCLI 真的能跑通：

```bash
opencli hackernews top --limit 5
```

结果能正常返回结构化数据。

这个步骤看似简单，但很关键。它证明了：CLI → daemon → Chrome 扩展 → 网站 → 结构化输出，这条链路是通的。

### 第三步：侦察内部白名单页面

接下来才进入真正的目标：内部白名单系统。

我先让 OpenCLI 打开页面：

```bash
opencli browser --session recon open "<internal-url>"
```

然后用 `state` 看页面结构。很快就发现页面其实是一个典型的后台 SPA，左侧菜单、右侧表格、搜索框、弹窗都能被 OpenCLI 识别出来。

更惊喜的是，页面列表已经渲染到了 DOM 里。也就是说，即使暂时找不到 API，也可以通过页面内容抽取拿到表格数据。

但只靠 DOM 抓取不够稳定。真正做适配器，最好还是找到后台接口。

于是我用浏览器里的 Performance API 看了一下 XHR 请求，找到了几个关键接口：

```text
selectStrategy          # 查询白名单策略列表
getStrategyDetail       # 查询策略详情
getWhitelist            # 查询某个策略下的账号列表
addWhiteList            # 添加账号到白名单
```

这些名字是我脱敏后的表达，真实接口名和路径这里就不写了。

### 第四步：实现 4 个只读能力

先从只读能力开始，这是我这次实践里最稳的部分。

我做了 4 个命令：

```bash
# 1. 查询我负责的白名单
opencli internal-whitelist mine

# 2. 搜索白名单策略
opencli internal-whitelist search --key "keyword"

# 3. 查看某个白名单下的账号列表
opencli internal-whitelist accounts --id <id>

# 4. 查看白名单策略详情
opencli internal-whitelist detail --id <id>
```

这些命令背后其实就是浏览器上下文里的 API 调用。由于它复用了浏览器登录态，所以不需要额外申请 token，也不需要把密码写进脚本。

这里踩了一个坑：OpenCLI 自带的 `fetchJson` 在这个页面上会和前端 bundle 里的某个全局变量冲突，报了一个很奇怪的错误。

最后的解决方案是：不用 `fetchJson`，而是在浏览器上下文里执行一个 async IIFE：

```js
(async () => {
  const r = await fetch('/api/path', {
    method: 'POST',
    credentials: 'include',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(payload)
  })
  return await r.json()
})()
```

这个方式比较朴素，但很可靠。

### 第五步：给写操作加上保护

最敏感的是"添加账号到白名单"。

我不想让 AI 直接上来就写数据。所以这个命令设计成了两段式：

```bash
# 默认 dry-run：只检查，不写入
opencli internal-whitelist add-account --id <id> --account <account-id>

# 明确确认后才写入
opencli internal-whitelist add-account --id <id> --account <account-id> --confirm
```

执行逻辑也很简单：

1. 先查询目标白名单已有账号列表
2. 如果账号已经存在，直接 `skip`
3. 如果账号不存在，且没有 `--confirm`，只提示会写入但不执行
4. 只有账号不存在且显式传了 `--confirm`，才调用写接口

实际测试时，我用的是一个已经存在的测试账号，所以结果是：

```text
account      action   status    message
<redacted>   skip     existed   已在白名单中
```

这说明去重逻辑生效了，也避免了重复写入。

## 💭 这次实验让我重新理解了 CLI

以前我理解 CLI，更多是给人用的命令行工具。

但这次感觉不太一样。OpenCLI 这类工具真正有意思的地方，是把 CLI 变成了 **AI Agent 的操作接口**。

传统浏览器自动化有几个问题：

- 页面结构不稳定，按钮位置一变就容易挂
- AI 每次都要重新理解页面，成本高
- 输出不稳定，很难接到后续工作流

而 CLI 的好处是：

- 输入参数固定
- 输出结构固定
- 可以校验、可以测试、可以复用
- AI 不需要每次重新"看网页"

比如我现在只要说：

> 看一下我负责的白名单里，哪些快过期了。

AI 就可以调用：

```bash
opencli internal-whitelist mine --format json
```

然后在 JSON 里按 `days_left` 过滤，最后给我一个总结。

这比让 AI 打开网页、读页面、手动找表格稳定太多。

## ⚠️ 也不是万能的

这次实验也暴露了几个现实问题。

### 1. 内部后台页面很脆弱

很多后台系统的前端实现比较复杂，接口命名、参数结构都不公开。要写适配器，还是要花时间侦察。

这次我是通过：

- 页面结构抽取
- Performance API
- 前端 bundle 关键词搜索
- 手动触发一次操作

才把接口摸清楚。

这不算优雅，但很实用。

### 2. 写操作要非常谨慎

查询类能力很适合 CLI 化。但写操作不一样。

尤其是内部系统里的写操作，可能会触发审批、变更、通知，不能因为 AI 能点按钮就放心交给它。

所以我的原则是：

- 默认 dry-run
- 明确 `--confirm`
- 先做去重检查
- 输出里明确告诉我到底执行了什么

这点比功能本身更重要。

### 3. 适配器会随页面变化失效

后台页面一改版，接口或字段可能就变了。

所以这类工具不能当成一次性脚本，而应该像小产品一样维护：有命令说明、有验证命令、有使用文档、有失败时的排查方式。

这次最后落地了 5 个能力：

| 能力 | 状态 | 价值 |
|---|---|---|
| 查询我负责的白名单 | 已完成 | 快速盘点责任范围 |
| 搜索白名单策略 | 已完成 | 按业务名称/KEY 查找策略 |
| 查看账号列表 | 已完成 | 回答"某账号是否已开白" |
| 查看策略详情 | 已完成 | 看负责人、状态、过期时间 |

从时间上看，这次大概花了一个晚上。

如果只算查询能力，真正能节省的不是几分钟点击时间，而是**减少切换上下文**。我不用从 IDE 跳到浏览器，再从浏览器跳回文档；AI 可以直接在当前上下文里查完、总结、归档。

这对产品经理挺有价值。

## 🤔 我觉得这个方向适合什么场景

经过这次实验，我觉得 OpenCLI 很适合下面几类场景：

### 适合

- 高频、固定流程的后台查询
- 有浏览器登录态但没有公开 API 的系统
- 输出能结构化的列表/详情页
- 需要和 AI 工作站、文档、周报串起来的流程

### 不太适合

- 一次性、低频操作
- 高风险写操作
- 页面经常变、流程不稳定的系统
- 强合规审批链路里的关键动作

换句话说，它更适合做"产品经理的浏览器副驾驶"，而不是完全替代人。

## 🎉 总结

这次实验对我最大的启发是：

> AI 提效不一定要从大系统开始。很多时候，把一个浏览器里的小后台操作变成稳定 CLI，就已经很有价值。

它不酷炫，也不复杂，但是真的贴近日常工作。

我以前做了很多文档类、知识库类、策划类的 AI 工作站能力；这次算是第一次把 AI 的手伸进浏览器后台里。虽然只做了一个白名单场景，但方向是成立的。

接下来我可能会继续尝试两个方向：

1. 把更多只读后台查询做成 CLI，比如商品状态、折扣配置、客户配置查询
2. 对写操作保持克制，只保留 dry-run + 人工确认的模式

最后再强调一下：本文所有内部系统名称、链接、账号和具体数据都已脱敏，示例命令也做了泛化处理。
