# 给产品经理做个"原型 Skill"：让 AI 替代 PPT 做产品汇报

## 🤔 起因：一个看似普通的产品问题

事情起源于一个特别普通的场景——我在跟 AI（CodeBuddy）聊我们产品里的一个概念叫 "Agent Engine"。

我们团队最近在讨论 Agent Runtime 的产品分层，内部推演会议定下来"上层拆成 Agent Engine + Agent Govern 两个平级产品"。但聊着聊着，我和 AI 都开始觉得这个定义很模糊：

> "Agent Engine 和 Agent Govern 到底是产品，还是只是一个品牌或者一个归纳而已？"

这个问题本来是想请 AI 帮我理清思路。然后 Google 在 2026-04-22 发布了 **Gemini Enterprise Agent Platform**，正好把类似的问题摆在了台面上——他们选择了"单一伞品牌 + 内部四大能力板块"的模式，和我们之前设想的"双产品平级"完全不同。

**于是这个讨论就滚动了起来。**

---

## 💭 讨论过程：把模糊的定义聊清楚

我和 AI 的对话大概经历了这几个阶段：

**第一阶段：质疑**

我一开始让 AI 只是"介绍一下 Agent Engine"。AI 结合工作站里已有的材料，发现工作站里实际上有**两个 Agent Engine** 的概念——Google 的 Vertex AI Agent Engine（竞品）和我们自己规划的 Agent Engine（产品线）。这个混淆一摆出来，问题就显现了。

**第二阶段：对标**

Google 新发布的 Agent Platform 一对照，差异非常明显：
- Google 用**单一伞品牌** → 我们原设计是 "no umbrella brand"
- Google 有完整的 **Build → Run → Govern → Optimize** 四段闭环 → 我们缺 Optimize
- Google 绑定 GCP → 我们有**三形态**（公有云/司内/私有化）

这一对比，反而让我对自家产品的独特价值更清楚了。

**第三阶段：做决策**

聊到一半，我有一个关键判断——**Agent Engine 只服务"有自研能力的企业内部 Agent 团队"**，不服务 Agent 服务商（那些需要的是原子 API，不是打包平台）。这个切分一出来，之前的所有混乱都顺了：

- "no umbrella brand" 的真实含义不是"不要统一品牌"，而是 **"不要把两类客户塞进同一个伞品牌"**
- Agent Runtime 是对外唯一商业化入口
- Engine / Govern / 原子能力是**内部**三条独立产品线

又聊了两轮，又定下了两个关键决策：
- **有独立 PM，但不做独立商业化**（学 AWS Bedrock 模式）
- **客户只买 Agent Runtime 一个产品**，Engine 作为"能力板块"在控制台露出

**聊完一看，已经有 8 项结构化决策了。**

AI 把这些决策整理成了一份正式的产品定义文档 `agent-engine-product-definition-v0.1.md`。

---

## 🎨 拐点：一句话改变了后续方向

文档出来之后，我的真实感受是：**"这个东西我怎么给老板讲？"**

8 页文档不可能让老板读完；做 PPT 又觉得太碎、太formal、而且画架构图特别费时间。我当时随手问了一句：

> "按这个定义，画一个静态html原型示意图出来，给老板看看我理解的对不对。"

AI 很快做了一个单文件 HTML，双击就能打开。打开那一刻我有点惊喜——**这比 PPT 好太多了**。

- 核心叙事直接放在 Hero 区
- 产品分层架构图用 CSS 盒子画出来，能看懂产品结构
- 8 项决策用表格 + 高亮标记
- 业界对标用卡片形式，一眼能看出我们和 Google / AWS / 阿里云的区别
- 整个页面滚动下去像一篇"产品故事"

我当时就说：**"我很喜欢这种模式，比 PPT、比文档汇报，清晰多了。帮忙总结并变成一个 Skill。"**

---

## 🛠️ 沉淀：从一次性产出到可复用 Skill

### Skill 是什么

简单说，在 CodeBuddy 里 Skill 就是**给 AI 设定的一套专业工作流**——遇到特定的触发词，AI 就按既定的 SOP 执行。比如我之前做过的：

- 摄取原始资料的 Skill
- 生成竞品调研报告的 Skill
- 维护腾讯云文档的 Skill

这次要做的叫 `html-prototype-report`——**"用交互式 HTML 原型替代 PPT/Word 做产品汇报"**。

### Skill 的核心设计

我和 AI 讨论后，明确了 Skill 的几个关键点：

**1. 两种模式先选对**

| 模式 | 何时用 | 产物 |
|------|--------|------|
| **A · 产品定位** | 讲"这个产品是什么" | 一页式滚动 HTML |
| **B · 控制台 UI** | 讲"这个产品长什么样" | SPA 多页切换 |

**2. 硬性规定**

- 单文件 .html，**双击可打开**
- **零外部依赖**：不引用 CDN、不加载外部字体
- **不用 React/Vue/Tailwind**，原生 HTML/CSS/JS 就够
- 视觉要像真实产品控制台（而不是"学生作业"）

**3. 6 步执行 SOP**

读源文档 → 选模式 → 提取骨架 → 写单文件 → 加交互（模式 B 必须）→ 归档 + 自动打开预览。

**4. 质量自检清单**

包括通用 + 模式 A + 模式 B 三类清单，每次生成完自查一遍。

## 🧪 试用：给 Memory 产品定义也做一个

Skill 写完之后，我立刻让 AI 试用——把 Agent Memory 这个产品（也是前几天刚做的产品定义）做成一个原型。

结果非常顺利。AI 产出了一个 11 章节的原型：

1. **Hero 区**：核心叙事 "不是卖智能，是卖可控的智能"
2. **ToC vs ToB 对比**：为什么企业需要另一种 Memory
3. **可选模块**：为什么 Memory 不是 AGS 默认能力
4. **三记忆类型**：事实 / 事件 / 程序，对应认知科学三分法
5. **统一记忆模型**：能力 + 治理的融合设计
6. **治理五维**：可见 / 可控 / 安全 / 合规 / 可靠
7. **三层交互**：API + 声明式 + 控制台
8. **人 × 系统 × Agent 的体系构建模型**
9. **8 项核心决策表**（D6/D8 高亮）
10. **9 家竞品格局**（我们的卡片有视觉强调）
11. **16 项开放问题**（P0-P3 分级）

视觉色调还是深色氛围（和 Agent Engine 原型风格一致），但用了"记忆青 + 三记忆色"的专属配色，既有系列感又有差异化。

**这次试用让我确认了一件事：Skill 的 SOP 是稳定的。** 不是偶然产出一个好原型，而是确实可以复用。

## 🔁 复用：把 Skill 扩展到另外两个工作站

这是整个过程最有意思的部分。

我手头有三个 AI 产品工作站：

- **Agent Runtime 工作站**（Agent 产品策划）
- **TKE Housekeeper 工作站**（TKE 产品策划）
- **TCR 产品设计工作站**（TCR 产品策划）

做完 Skill 后我立刻想到：**"这个能力不应该只在 Agent Runtime 里用"**。TKE 和 TCR 的产品也天天需要做汇报啊。

但这里有个问题：第一版 Skill 里有不少 Agent Runtime 工作站特有的东西——

- 硬编码了 `docs/03-planning/prototypes/` 路径
- 写死了 `VAULT-INDEX.md` 作为活动日志
- 默认配色是 Agent 领域的紫 / 青
- 样例引用了 Agent Engine 产品

### v1.0 → v2.0：升级成通用版本

我让 AI 先把 Skill **脱敏化**，变成跨工作站通用的 v2.0 版本：

| v1.0（Agent Runtime 专用） | v2.0（通用） |
|------------------------|-------------|
| 硬编码输出路径 | 新增 **Step 0 工作站适配**，自动识别约定 |
| 硬编码 VAULT-INDEX | 检测活动日志文件，没有就跳过 |
| 默认领域配色 | 内置领域配色表（TKE 用 K8s 蓝 / TCR 用腾讯云蓝） |
| 路径示例写死 | 去掉具体路径引用，改为跨工作站通用示例 |
| 骨架建议"6-8 章" | 改为"**核心 6 章必选 + 按文档密度追加 3-5 章**"（基于 Memory 11 章的实战经验反馈） |

### 一键复制到两个工作站

升级完成后，我告诉 AI 要复制到 TKE Housekeeper 和 TCR 产品设计两个工作站。AI 做了几件事：

1. 读两个工作站的 `CODEBUDDY.md`，确认目录约定（发现都有 `docs/03-planning/` 这个结构）
2. 检查 `.codebuddy/skills/` 目录（两个都已存在）
3. 复制 Skill 文件（同一份 v2.0 内容）
4. 为 TCR 创建 `prototypes/` 目录（TKE 已经有了，里面甚至还躺着几份我之前手动做的 HTML）
5. 给两个工作站都建了 `prototypes/README.md` 作为原型库索引
6. 在两个工作站的 `CODEBUDDY.md` 里登记这个新 Skill

最后提交到工蜂——三个仓库同时推送，commit message 统一用中英混合风格。

**整个过程从"让 AI 总结成 Skill"到"三个工作站都能用"，大概 15 分钟。**

## 💭 真实感受和反思

### 什么做得好

**1. "先解决具体问题，再沉淀工具"的顺序是对的**

我不是一上来就说"我要做个原型 Skill"。是先有了一个真实的产品问题（Agent Engine 定位模糊），在解决它的过程中产出了原型，然后才沉淀成 Skill。

这个顺序保证了 **Skill 是从真实场景中长出来的**，不是拍脑袋想出来的。

**2. 跨工作站复用的决策很值得**

如果我只在一个工作站做这件事，下次换到 TKE 场景还得重来。花 10 分钟把 Skill 通用化，换来三个工作站永久可用——这笔账很划算。

**3. AI 帮我减少了很多"重复执行"的工作**

提交到工蜂这种事，我自己做要：检查 git status → stage → 写 commit message → push → 三个仓库重复三次。让 AI 做就是并行一次搞定。

### 什么不够好

**1. 原型的视觉精细度还有限**

虽然比 PPT 好，但真正拿去给老板汇报，可能还需要再调一调配色、间距、字体。AI 生成的版本是"能用"，还达不到"惊艳"。

**2. Skill 的通用化不是银弹**

v2.0 里我加了 "Step 0 工作站适配"，理论上能在新工作站自动识别环境。但实际跑起来，还是偶尔需要我明确告诉 AI "这是 TKE 工作站，请用 K8s 蓝做主色"。自动化没做到 100%。

**3. 原型的信息密度需要把握**

Memory 产品定义做出 11 章的原型，老板可能还是会觉得太多。其实更好的版本可能是**10 分钟精简版**（只保留核心 5 章），但这需要我后续再迭代 Skill 的指导原则。

## 🤔 给其他人的建议

**1. 不要一开始就想"做一个通用工具"**

我见过很多人上来就说"我要做一个通用的 XX 平台"，然后花大量时间在抽象和架构上。结果做出来的东西很复杂，但没人真的在用。

我这次的顺序是：**一次具体的需求 → 产出一个解决方案 → 发现可以复用 → 再抽象成 Skill**。这个顺序下做出来的工具，至少能保证自己会用。

**2. PPT 不是汇报的最优解**

尤其是讲**产品定位、架构、策略**这种东西时，PPT 的翻页结构反而会把叙事切碎。一个 HTML 页面滚动下来，有完整的叙事流，老板更容易跟得上。

我不是说 PPT 没用，而是**产品经理手里应该有多种汇报武器**——PPT、文档、HTML 原型、视频、演示 demo，哪个最合适就用哪个。

**3. AI 协作的关键是"给它一个稳定的 SOP"**

单次让 AI 做一个原型，每次结果可能不一样。但如果你把"做原型"这件事拆成 6 步 SOP，加上质量自检清单，AI 的输出就会非常稳定。这就是 Skill 的价值——**把一次成功的经验，变成可复现的流程**。

## 🎬 总结

这篇博客其实记录的是一个很小的动作：**把一次和 AI 的成功协作，沉淀成了一个可复用的 Skill，并扩展到多个工作站。**

但这个动作背后是一个我越来越相信的工作方式：

> **产品经理 + AI 的协作，不应该是"每次都从零开始"，而是"让好的经验沉淀下来、长期复用"。**

每次让 AI 做一件新事情，都是一次投资机会——要么这次做完就忘了，要么把它变成能反复用的工具。

Skill 就是这个"变成工具"的载体。

下一步我想试试把其他重复做的事也 Skill 化——比如"竞品对比分析"、"产品路线图生成"、"客户访谈纪要整理"等等。希望几个月后回头看，我手里已经有一套成体系的"AI 产品经理工具箱"。
