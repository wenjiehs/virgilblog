# 用 CodeBuddy 搭建产品策划工作站：一个产品经理的 AI 协作实验

## 🤔 为什么要做这件事

作为 TKE Housekeeper 的产品经理，我日常的工作主要包括：

- **功能策划**：写各种 PRD、功能策划稿
- **产品知识管理**：整理产品逻辑、设计原则、功能说明
- **数据分析**：客户损益分析、成本收益计算
- **竞品调研**：对比分析、差异化定位

这些工作有个共同特点：**重复性高、格式固定、但又需要大量上下文**。

比如写一份功能策划稿，结构基本都是：背景动机 → 目标用户 → 功能设计 → 技术方案 → 实施计划。但每次都要从头写，还要翻之前的文档找产品逻辑、设计原则等信息。

我就在想：**能不能让 AI 记住我的产品逻辑，然后按照固定模板帮我生成策划稿？**

这个想法促使我开始探索 CodeBuddy 的「规则」「代理」「技能」机制。

---

## 🎯 我想实现什么

一句话：**把产品策划变成"对话"**。

具体来说，我希望实现这样的交互：

```
我：我想给 TKE Housekeeper 做一个安全模式

AI：好的，我来帮你生成策划稿...
    [自动分析需求]
    [查阅产品知识库]
    [按模板生成策划稿]
    [进行自我评审]
    
✅ 策划稿已生成：docs/03-planning/feature-proposals/2026-03-10_安全模式.md
```

**而不是**：

```
我：帮我写一个安全模式的策划稿

AI：好的，请先告诉我：
    1. 安全模式的背景是什么？
    2. 目标用户是谁？
    3. 核心功能有哪些？
    4. 技术方案怎么考虑？
    5. ...（一堆问题）

我：[崩溃]
```

关键区别在于：**AI 应该主动利用已有的产品知识，而不是每次都从零开始问我**。

```

├─────────────────────────────────────────────────────────────┤
│                                                             │

│  │                    规则层 (Rules)                    │   │

│  │  │ ─────────── │ │ ─────────── │ │ ─────────── │   │   │
│  │  │ 功能策划     │ │ 损益分析     │ │ 逻辑归档     │   │   │
```

### 为什么是三层？

**规则层**：定义"怎么做"
- 策划稿应该包含哪些章节？
- 文档格式规范是什么？
- 评审标准是什么？

**代理层**：定义"谁来做"
- 产品架构师负责功能策划
- 数据分析师负责损益分析
- 知识管理器负责逻辑归档

**技能层**：定义"具体做什么"
- 按模板生成策划稿
- 按公式计算财务指标
- 按规范生成报告

这样设计的好处是**职责清晰、可复用、易维护**。

## 🛠️ 实际动手过程

### 第一步：创建项目结构

首先，我在本地创建了工作站的目录结构：

```
TKEHousekeeper产品设计/
├── README.md                    # 工作站说明
├── TODO.md                      # 迭代计划
├── CHANGELOG.md                 # 变更日志
├── CODEBUDDY.md                 # CodeBuddy 指导文件
│
├── docs/                        # 产品文档
│   ├── 01-product-concepts/     # 产品理念
│   ├── 02-features/             # 功能模块
│   ├── 03-planning/             # 产品策划
│   │   └── feature-proposals/   # 功能提案
│   │       └── _template.md     # 策划稿模板
│   ├── 04-analysis/             # 客户分析
│   │   └── cost-benefit/        # 损益分析
│   │       └── _template.md     # 分析报告模板
│   └── 05-knowledge-base/       # 知识库
│
└── .codebuddy/                  # CodeBuddy 配置
    ├── rules/                   # 工作站规则
    ├── agents/                  # 工作流代理
    └── skills/                  # 技能包
```

这个结构的核心思路是：**让文档按用途分类，让 AI 知道去哪里找信息**。

### 第二步：编写 CODEBUDDY.md

这是整个工作站的"说明书"，告诉 AI 这个项目是干什么的、怎么工作。

我在里面详细描述了：

1. **项目定位**：这不是代码项目，而是产品策划工作台
2. **核心能力**：产品逻辑归档、策划稿生成、损益分析
3. **工作流映射**：什么场景触发什么代理
4. **文档结构**：每个目录放什么内容

这一步非常关键，**CODEBUDDY.md 决定了 AI 对项目的理解**。

### 第三步：设计策划稿模板

我参考了公司内部的 PRD 模板，设计了一个通用的功能策划稿模板：

```markdown
# 功能策划稿模板

## 📋 基本信息
- 功能名称、提案人、创建日期、状态、优先级

## 🤔 背景与动机
- 问题描述、目标用户、使用场景、核心价值

## 🎯 产品目标
- 核心目标（3-5个）、成功指标（可量化）

## 💡 功能设计
- 核心功能详细设计、用户操作流程、界面交互

## 🔧 技术方案
- 技术架构、核心技术、数据模型、API设计

## 🚀 实施计划
- 开发阶段、里程碑、负责人、状态

## ⚠️ 风险与依赖
## 💰 成本效益分析
## 📊 市场分析
## ✍️ 评审记录
```

模板的设计原则是：**结构完整但不繁琐，必填项清晰，选填项灵活**。

### 第四步：配置产品架构师代理

这是最核心的一步。我需要告诉 AI 如何扮演"产品架构师"的角色。

在 `.codebuddy/agents/product-architect.md` 中，我定义了：

**1. 触发条件**

```markdown
当用户满足以下任一条件时，自动触发本代理：

1. 提出产品想法：
   - "我想实现..."
   - "我们需要..."
   - "建议增加..."

2. 描述功能需求：
   - "希望能支持XX功能"
   - "需要XX这样的能力"

3. 提出痛点：
   - "现在的问题是..."
   - "用户反馈XX不好用"
```

**2. 工作流程**

```markdown
阶段1: 需求理解与分析
  → 提取核心需求、目标用户、使用场景、预期效果

阶段2: 策划稿生成
  → 使用模板生成策划稿各章节

阶段3: 自我评审
  → 对照评审清单检查完整性和质量

阶段4: 输出交付
  → 保存文件、输出摘要、提供下一步建议
```

**3. 评审清单**

```markdown
内容完整性：
- [ ] 所有必填章节都已填写
- [ ] 背景与动机清晰
- [ ] 产品目标明确可量化
- [ ] 功能设计完整

质量标准：
- [ ] 目标符合SMART原则
- [ ] 技术方案可行
- [ ] 里程碑可追踪
```

### 第五步：配置知识管理器代理

知识管理器负责"记住"我告诉它的产品逻辑。

它的工作流程是：

```
用户输入产品逻辑 
  → 信息提取（识别概念名称、定义、背景、价值）
  → 知识归类（判断归档到产品理念还是功能模块）
  → 结构化归档（使用标准文档结构保存）
  → 关联建立（创建与其他概念的链接）
  → 索引更新（更新知识库索引）
```

比如，当我说：

> "TKE Housekeeper 的动态调度能力包含三个核心特性：均衡负载调度、提升装箱调度、规整业务调度..."

知识管理器会自动：
1. 识别这是"功能逻辑"类型的知识
2. 归档到 `docs/02-features/resource-scheduling/dynamic-scheduling.md`
3. 在知识库索引中添加条目

## 💭 实际使用效果

### 场景 1：功能策划

我输入：

> "我想给 TKE Housekeeper 做一个 AI Native 体验增强，让 AI 生成的应用在 K8s 上更好运行、部署、运维"

AI 自动执行：
1. 分析需求核心：面向 AI 时代的产品体验升级
2. 识别目标用户：AI 编程工具用户、Vibe Coding 实践者、AI Agent 开发者
3. 规划产品方向：TKE MCP Server、Agent CLI、CodeBuddy Skill、AI 应用运维增强
4. 生成策划稿：39KB 的详细策划文档

输出：`docs/03-planning/feature-proposals/2026-03-10_AI-Native体验增强.md`

**生成的策划稿包含**：
- 详细的背景分析（行业趋势、当前痛点）
- 清晰的目标用户画像
- 具体的使用场景
- 4 个产品方向的详细设计
- 成功指标和实施计划
- 技术架构图和 API 设计

### 场景 2：产品逻辑归档

我把官方文档中的产品介绍丢给它：

> "帮我归档一下原生节点的概述..."

AI 自动：
1. 提取核心概念（原生节点定义、核心价值、工作原理）
2. 归档到 `docs/01-product-concepts/native-nodes.md`
3. 更新知识库索引

后续策划时，AI 会自动引用这些归档的知识。

### 场景 3：知识库查询

当我策划新功能时，AI 会主动查阅已归档的产品知识：

> "基于已归档的 FinOps 成本优化理念，这个功能可以复用动态调度能力的装箱算法..."

这就是我想要的效果：**AI 真正"记住"了产品逻辑，而不是每次都问我**。

## 🤔 一些思考

### 什么情况下适合搭建工作站？

1. **有固定的工作流程**：比如策划稿、分析报告这种格式固定的输出
2. **需要大量上下文**：AI 需要了解产品背景、设计原则等信息
3. **重复性高**：类似的工作会反复做

### 工作站 vs 普通对话

| 普通对话 | 工作站 |
|---------|--------|
| 每次从零开始 | 有持续积累的知识库 |
| 格式随机 | 严格遵循模板 |
| 需要反复确认 | 代理自主决策 |
| 适合一次性任务 | 适合长期重复任务 |

### 目前的局限

1. **初始配置成本高**：需要花时间设计规则、代理、模板
2. **知识归档仍需人工**：AI 提取的信息需要人工确认
3. **复杂场景处理有限**：涉及多个代理协作时还不够智能

### 后续可以做什么

- 增加自动竞品分析能力
- 增加自动市场调研能力
- 优化多代理协作机制
- 提升知识图谱构建能力

## 🎉 总结

用 CodeBuddy 搭建产品策划工作站，本质上是**把产品经理的工作方法论固化成 AI 可执行的规则**。

核心收获：

1. **CodeBuddy 的三层架构（规则、代理、技能）非常适合构建复杂的工作流**
2. **CODEBUDDY.md 是关键**，它决定了 AI 对项目的理解深度
3. **模板化 + 知识库 = 高质量输出**
4. **初期投入高，但后续效率提升明显**

如果你也有类似的重复性工作，不妨试试用 CodeBuddy 搭建一个专属工作站。
