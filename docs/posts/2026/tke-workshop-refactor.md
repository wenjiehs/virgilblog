# 一个产品经理的 AI 编程实验：重构 TKE Workshop

## 🤔 为什么要做这件事

作为一个产品经理，我一直在思考一个问题：**AI 时代，非技术背景的人能不能真正"编程"？**

恰好手头有个小需求：[TKE Workshop](https://tke-workshop.github.io/) 是我们团队维护的一个用户文档站点，内容更新全靠手动，体验也比较陈旧。我想试试能不能借助 AI 工具（主要是 Claude + Cursor）把它改造一下。

**坦白说，这更像是个玩具项目**——没有复杂的业务逻辑，也没有性能压力，纯粹是验证一个想法：产品经理能否通过 AI 从"需求描述者"变成"构建者"。

## 🎯 我想做什么

### 初始想法

原有的站点是静态 HTML，每次更新内容都要改代码、重新部署。我的想法很简单：

1. **自动抓取内容** - 能不能从 GitHub 仓库自动读取 README，省去手动复制粘贴？
2. **改改界面** - 统一一下风格，让它看起来没那么"古早"
3. **加点新内容** - 最近 AI Agent 很火，想试试能不能加点相关的实践案例

就这么简单。没想做什么"革命性产品"，就是想验证：**一个产品经理，能不能在 AI 的帮助下把想法变成现实**。

### 我的技术背景

说实话，我的前端水平大概停留在"能看懂 HTML/CSS，能改改 JavaScript"的程度。之前写过一些简单的内部工具，但离专业前端工程师还差得远。

这次的优势是：
- ✅ 会用 Git（基本操作）
- ✅ 懂一点 JavaScript 语法
- ✅ 知道产品要做成什么样

挑战是：
- ❌ 不太懂现代前端框架
- ❌ 没系统学过前端工程化
- ❌ 对 GitHub API 一知半解

**所以这次实验的核心问题是：这点技术背景 + AI 工具，能走多远？**

## 🛠️ 实际动手：一个月的迭代

### 第一周：搞定 GitHub 自动抓取

**需求**：网站能自动从 GitHub 仓库读取 README 并展示。

我对 Claude 说："我想从 GitHub 读取某个仓库的 README，然后显示在网页上，怎么做？"

Claude 给了我一个方案，核心就是调用 GitHub API：

```javascript
// 这是 AI 给我的代码（稍作调整）
async function fetchGitHubReadme(repoUrl) {
  const apiUrl = convertToGitHubAPI(repoUrl)
  const response = await fetch(apiUrl)
  const data = await response.json()
  return decodeBase64(data.content)
}
```

**遇到的坑**：
- 图片路径不对：README 里的相对路径在网页上显示不出来
- 链接跳转有问题：点击链接跳到了错误的地址

**怎么解决的**：
继续问 AI，它教我用正则表达式替换路径。我大概能看懂，但自己写不出来。

```javascript
// AI 帮我写的路径处理
normalizeImages(markdown, repoUrl) {
  return markdown.replace(
    /!\[([^\]]*)\]\((?!http)([^)]+)\)/g,
    `![$1](${repoUrl}/raw/main/$2)`
  )
}
```

**结果**：花了 2 天，终于能正常抓取和显示 GitHub 内容了。感觉很神奇——这在以前是我完全做不到的事。

### 第二周：改界面

**需求**：统一风格，让页面看起来更现代。

我不太懂 CSS，但知道自己想要什么样的效果。我给 AI 看了几个参考网站，说："我想要这种卡片式布局，鼠标悬停有动画效果。"

AI 给了我一段 CSS：

```css
.cookbook-card {
  background: linear-gradient(135deg, #667eea 0%, #764ba2 100%);
  border-radius: 12px;
  padding: 24px;
  transition: transform 0.3s ease;
}

.cookbook-card:hover {
  transform: translateY(-8px);
}
```

直接复制粘贴，效果还不错。然后慢慢调整颜色、间距，大概花了 3 天把整体风格统一了。

**体会**：我做的主要是"决策"和"微调"，具体代码大部分是 AI 生成的。这种协作模式挺有意思。

### 第三周：加点新内容

**想法**：最近 AI Agent 挺火的，能不能加个 "Agent Sandbox" 分类？

这部分主要是内容工作，我自己写了 6 个简单的 Cookbook，主题是"如何在 TKE 上跑 AI Agent"。技术上没什么难度，就是整理知识。

同时让 AI 帮忙：
- 调整导航结构，加上新分类
- 优化首页布局，突出新内容
- 把原来的"学习路径"改名为"Agent-Ready Best Practices"（听起来更酷）

### 第四周：收尾和优化

最后一周主要是：
- 修 bug（AI 写的代码也会有问题）
- 加缓存（不然频繁调 GitHub API 会被限流）
- 写点文档
- 清理冗余代码

**最终成果**：提交了 13 个 PR，网站能用了，虽然还有很多不完美的地方。

## 💭 一些真实的感受

### AI 能做什么，不能做什么

**AI 擅长的**：
- ✅ 写标准化的代码（API 调用、数据处理、样式代码）
- ✅ 解释概念（比如什么是 CORS，怎么处理异步）
- ✅ 提供多个方案（给出 2-3 种实现方式让我选）
- ✅ 调试问题（告诉它报错信息，它能分析原因）

**AI 不擅长的**：
- ❌ 理解你的真实意图（如果需求描述不清，它也会迷茫）
- ❌ 复杂的架构设计（涉及多个系统交互时容易出错）
- ❌ 性能优化（它给的方案能用，但不一定最优）
- ❌ 理解业务逻辑（你得自己想清楚产品要做成什么样）

### 产品经理的角色变化

以前我的工作流程：
1. 想清楚需求
2. 写 PRD 文档
3. 和开发沟通
4. 等待开发实现
5. 验收测试

现在借助 AI：
1. 想清楚需求
2. ~~写 PRD~~ 直接对 AI 描述需求
3. ~~和开发沟通~~ 和 AI"沟通"
4. ~~等待开发~~ 自己迭代调整
5. 验收（自己验自己）

**变化是**：从"需求定义者"变成了"需求定义者+实现者"。虽然技术深度还不够，但起码能把想法快速落地了。

### 这个项目的价值在哪

说实话，TKE Workshop 现在还只是个"能用"的状态，离真正的工程化项目差得远：
- 没有完善的测试
- 没有错误监控
- 性能优化也很粗糙
- 代码质量参差不齐

但对我来说，价值在于：

1. **验证了一个假设**：产品经理确实可以通过 AI 从 0 到 1 搭建一个可用的项目
2. **学到了很多东西**：虽然很多代码是 AI 写的，但我起码知道了现代前端是怎么回事
3. **改变了思维方式**：以后写需求时，会更理解开发同学的难处
4. **建立了信心**：原来我也可以"编程"

**它不是一个完美的产品，但它是一个有意义的实验。**

## 🤔 给其他产品经理的建议

如果你也想尝试"AI 编程"，我的经验是：

### 1. 从小项目开始

不要一上来就想做复杂系统。找个小需求（比如内部工具、文档站点），先把流程跑通。

### 2. 学会提问

AI 不是魔法，它需要你提供清晰的上下文。好的提问方式：
- ❌ "帮我写个网站"
- ✅ "我想做一个展示 GitHub README 的页面，用 JavaScript fetch API 调用 GitHub API，你能给我示例代码吗？"

### 3. 理解代码逻辑

你不需要会写，但起码要能看懂。不然 AI 给你的代码出了问题，你根本不知道怎么改。

### 4. 接受不完美

AI 生成的代码能用，但不一定优雅。如果你的目标是"快速验证想法"，那就够了。如果要上生产环境，还是需要专业开发review。

### 5. 保持好奇心

这个过程很有趣，你会学到很多东西。即使最后项目失败了，这些知识也是有价值的。

- **主要技术栈**：HTML/CSS/JavaScript + GitHub Pages

## 🎉 总结

这次实验让我确信：**AI 时代，产品经理真的可以成为"构建者"**。

你不需要成为专业开发，但你可以：
- ✅ 把想法快速原型化
- ✅ 理解技术实现的复杂度
- ✅ 和开发团队更好地沟通
- ✅ 独立完成一些小工具/小项目

TKE Workshop 还是个玩具项目，但它证明了一件事：**技术门槛正在降低，想法的价值正在提升**。

如果你也是产品经理，不妨试试看。说不定，你也能做出点有意思的东西。
