# 工作站的下一步：从 Human-in-the-Loop 到 Human-on-the-Loop

## 发现了一个瓶颈

今天在工作站里忙了一整天——给 TCR 的知识库新增了 20 多篇深度文档，把 Skills Registry 从一个功能扩展规划成了独立产品，顺手还重新设计了 CHANGELOG。

忙完之后突然意识到一件事：这些工作全都是我主动触发的。

- 想知道 TCR 按需加载的技术细节？要我说"继续学习"
- 想整理 Skills Registry 的交互设计？要我说"展开设计"
- 想做竞品巡检？要我说"更新竞品信息"

本质上，**我是整个系统的瓶颈**。工作站的所有工作流，都需要经过我这个入口才能启动。

这就是 Human-in-the-Loop 的局限——人既是触发器，又是调度者，还是最终的验收者。工作站越强大，我反而越忙。

---

## Human-in-the-Loop vs Human-on-the-Loop

这两个概念听起来像绕口令，但区别其实很直观：

**Human-in-the-Loop（人在循环中）**：
```
你发起 → AI 执行 → 你看结果 → 你再发起 → AI 再执行 ...
```

**Human-on-the-Loop（人在循环上）**：
```
AI 自主运转 → 积累产出 → 你定期查看、决策、拍板
```

前者你是发动机，后者你是方向盘。

前者的问题很明显：你不说话，什么都不发生。周末不开电脑，竞品出了大新闻，工作站毫不知情。你忙着开会，上周积累的客户需求没有人整理，就散落在各种对话记录里消失了。

后者的目标是：工作站持续运转，你只在关键节点介入——优先级决策、内容审核、方向确认。

---

## 我规划的三层演进

想清楚问题之后，我把演进路径拆成了三层。

### 第一层：让 Agent 定期主动工作

最简单的起点是定时任务。不需要复杂的 Agent 协作，只需要让现有的 Agent 按时间表自动触发。

**竞品动态周报**（每周一 09:00）

Agent 自己去扫各家竞品的官方博客和 What's New 页面，把新发布整理归档，顺手在竞品索引里记一条本周情报摘要。

周一上班，打开文件，5 分钟看完上周竞品有什么动静。从"我需要主动去查"，变成"它主动推给我"。

**知识库健康巡检**（每月月初）

自动扫描哪些文档标记了"待归档"但一直没动，哪些文档超过 60 天没更新，生成一份待办清单。每月初花 10 分钟看一眼，决定优先补哪块内容。

**需求状态周报**（每周五）

汇总本周新增需求、状态变更，输出到 `07-customer-requirements/weekly-report/`。周五下班前瞄一眼，形成一周的工作记忆。

这三个任务的设计原则是：**Agent 产出，人来决策**。Agent 不决定要不要做，只是把信息整理好呈现出来，决策还是在我这边。

### 第二层：Agent 之间传递任务

第一层是"定时触发"，第二层是"事件触发"——一个 Agent 的输出触发另一个 Agent 的工作。

几个具体场景：

**需求 → 策划自动流转**：`requirements-manager` 将需求标记为高优先级后，自动在任务队列里投递一条任务，`product-architect` 读取队列，生成策划稿草稿，完成后需求卡片状态自动更新。

**策划 → PR 稿接力**：策划稿完成后，自动触发 `content-creator` 生成对应的 PR 稿初稿，不需要我手动去说"帮我写一篇 PR 稿"。

**竞品差距 → 想法记录**：竞品巡检发现某个新能力，自动在想法收集区生成一条记录，下次做规划时自然会看到。

这一层的本质是：**减少我在 Agent 之间"传话"的工作**。

### 第三层：监控面板与主动通知

前两层解决"触发"和"联动"，第三层解决"可见性"——你不用一个个打开文件，一眼就能看到工作站的全局状态。

设想是一个 `workstation-status.md` 快照文件，由 Automation 定期维护：

```markdown
## 本周产出摘要
- 竞品巡检：AWS ECR 新增 1 条动态，其余无变化
- 新增需求：3 条（2 条待评估，1 条进入 backlog）

## 待你决策的事项
- [ ] 需求 REQ-20260401-001 优先级评估（Agent 已完成初评）
- [ ] 策划稿草稿 2026-04-01_xxx.md 待审核

## 异常提醒
- ⚠️ Harbor v2.11 发布，包含重大安全更新，建议关注
```

你只需要看这一个文件，就知道该做什么。

## 今天实际做了什么

想清楚这三层之后，我没有停留在规划阶段——今天直接把第一层落地了。

### 配置了竞品周报 Automation

在工作站里写好了 Automation 配置文件（`.codebuddy/automations/competitive-weekly/automation.toml`），任务内容是：

- 每周一 09:00 自动运行 competitive-researcher Agent
- 依次搜索 AWS ECR、Azure ACR、Google AR、Harbor、阿里云 ACR 五家竞品的近期动态
- 追加到对应竞品档案，更新竞品索引的周报摘要
- 如发现重大变化（新版本、重大功能、定价调整），在全局索引标注提醒

然后在另一个 IDE（WorkBuddy）里把 Automation 跑了起来。下周一就会自动触发第一次巡检。

### 写了演进规划文档

在 `workstation-guides/human-on-the-loop.md` 里把这三层演进路径写清楚了，包括每一步的具体任务清单、设计原则，以及"哪些始终需要人来判断"的边界说明。

这个文档算是工作站给自己立的路线图。

### 顺带更新了 CHANGELOG

之前的 CHANGELOG 只是文件列表，记录"新增了哪些文件"。今天重新设计成了双轨结构：

- **工作站能力轨**：新增了哪些 Agent、规则、Skill
- **TCR 业务知识轨**：归档了哪些产品逻辑、产出了哪些策划稿

这样回看的时候，能区分"工作站在进化"和"知识在积累"这两件不同的事。

## 今天还做了很多其他事

趁着说到工作站，顺便记一下今天整体做了什么——因为其实 Human-on-the-Loop 只是今天的最后一个想法，在这之前还做了不少事。

**TCR 知识库大扩充**

上午花了差不多两个小时，让工作站自主学习了大约 20 篇 TCR 官方文档，新增了按需加载、多地域复制、交付流水线、服务账号、镜像迁移、CI/CD 全链路等深度文档。`docs/02-features/` 从 6 篇扩充到了 26 篇。

这是典型的"我催促 Agent 学习"的工作模式——有点讽刺，今天正好在思考怎么让工作站自己学习。这种定期知识扩充，其实也很适合做成 Automation。

**Skills Registry 产品策划**

下午把 Skills Registry 从"TCR 的扩展功能"重新定位成了一个独立新产品。写了完整的产品策划稿，包括商业定价设计（四层版本体系）和产品交互设计（5 个核心页面 + 可演示的静态 HTML 原型）。

这个策划稿的有意思之处在于：在 brainstorming 阶段，AI 通过五个问题帮我把模糊的"想做个新产品"变成了清晰的"企业版 ClawHub，独立计费，v1 聚焦注册和治理，Runtime 对接做成插件"——这种结构化的想法提炼，是我觉得 AI 最有价值的使用场景之一。

---

## 一点感受

工作站用到现在，最深的感受是：**AI 工具的价值，很大程度上取决于你给它设计了什么样的结构**。

同样是"让 AI 帮你整理竞品"，一次性对话和有结构化知识库的工作站，效果天差地别。

但结构本身也会成为新的瓶颈。当工作站有了 7 个 Agent、26 篇功能文档、几十个策划稿和分析报告之后，光维护它就需要不少精力——这和"用 AI 节省精力"的初衷有点背道而驰。

所以 Human-on-the-Loop 对我来说不只是个技术方向，更是一种设计哲学：**工具应该适应人的工作节奏，而不是反过来要求人去伺候工具**。

第一层任务跑起来之后，看看实际效果。如果一个月后竞品档案在自动更新、知识库缺口在自动提醒，那这个方向就是对的。
