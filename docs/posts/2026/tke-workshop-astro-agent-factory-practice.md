# 把 Workshop 从文档站改成 Agent-First 工作坊：一次 Codex 协作复盘

这次改 Workshop，最开始并不是一个特别宏大的目标。

我只是觉得原来的站点体验有点割裂：文档是一套体验，Cookbook 又像另一套体验。前者更像传统知识库，后者更像一些散落的可执行脚本入口。用户如果只是想学某个云原生产品，可以顺着文档读；但如果想马上执行一个任务，比如创建集群、部署 Nginx、跑 GPU 工作负载，就要在站点、仓库和脚本之间来回跳。

更关键的是，AI Agent 时代的 Workshop 不应该只是"给人看的页面"。它还应该把上下文、输入条件、执行命令、验证步骤、风险提示、清理动作这些东西组织好，让人和 Agent 都能更稳定地使用。

所以这次我想做的其实是：把 Workshop 从一个文档站，改成一个更像"Agent-First 工作坊"的产品。

## 为什么放弃 MkDocs Material

一开始我还想在 MkDocs Material 上继续改。但做着做着会发现，它很适合传统文档站，却不太适合我想要的这种体验：

- 首页需要更像一个 Workshop 入口，而不是文档目录。
- 一级导航应该有明显的信息架构，而不是把所有内容都挤在侧边栏。
- Cookbook 不只是文章列表，它需要卡片、标签、风险等级、耗时、来源仓库和执行入口。
- 详情页需要复制按钮、参数表、费用提醒、清理步骤、相关文档回链。
- 未来还可能需要把更多执行层能力暴露给 Agent。

这些都能在 MkDocs 里硬做，但会越来越像和框架较劲。

后来我决定不把 MkDocs Material 保留为主体验，而是切到 Astro + Starlight。这个选择不是因为它更"潮"，而是因为它给了我一个更合适的边界：文档仍然可以是 Markdown，页面体验和数据结构则可以用组件来组织。

这对 Workshop 很重要。因为 Workshop 的内容不是单纯的"章节"，它更像一组可被浏览、筛选、执行、验证和复用的任务。

## Codex 这次像一个持续在场的工程同伴

这次主要是和 Codex 一起做。我给它的指令其实很口语化：先熟悉项目，再重构体验；不好就别保留 MkDocs Material；本地部署预览；提交远程；看到线上失败再修；截图里发现卡片堆叠，就继续改。

这个过程里，Codex 不只是生成代码。它做了几类很像工程协作的事情：

第一类是摸清现状。它先看项目结构、构建脚本、部署配置、旧文档和旧 Cookbook 内容，而不是直接重写页面。

第二类是把体验问题翻译成代码改动。比如"所有都放到一个目录里有点长，一级导航可以放到顶部吗？"这不是一个具体技术需求，但它背后是信息架构的问题。最后改成了顶部一级导航 + 侧边栏二级导航，让用户先选择 Start、基础操作、最佳实践、AI 实践、数据实践、Workshop Paths、Contribute 等大类，再进入具体内容。

第三类是持续验证。每次改完不是只说"好了"，而是跑测试、跑构建、开本地预览、看真实浏览器，再看 GitHub Actions 的构建和部署状态。

第四类是处理迁移过程里的反复。比如我后来发现"原来站点的文档和 cookbook 都不见了"，这说明迁移不只是搭一个新壳，还要把旧内容完整迁走。再比如 Cookbook 页面出现卡片堆叠，说明数据迁移完成了，但视觉密度和响应式布局还没过关。

这也是我这次最深的感受：AI 编程真正有价值的地方，不是一次性吐出很多代码，而是能在一个真实项目里跟着问题来回走。

## Cookbook 迁移：从脚本集合变成执行体验

这次最核心的部分是 Cookbook。

旧版 Cookbook 更像"有一些脚本和说明"。新版里，我希望它变成 Workshop 的执行层：每个 Recipe 都应该回答几个问题：

- 它要解决什么问题？
- 需要什么前置条件？
- 会产生什么费用或风险？
- 要输入哪些参数？
- 怎么复制命令并执行？
- 怎么验证执行成功？
- 怎么清理资源？
- 如果想理解原理，应该回到哪篇文档？

所以新版 Cookbook 不是简单把旧 Markdown 搬过来，而是把 17 个旧版 Recipe 都整理成结构化条目，包括：

- 创建托管集群
- 部署 Nginx 应用
- 部署 GPU 工作负载
- AI Playbook
- Chaos Engineering 混沌工程
- 集群直接升级
- 获取客户端真实 IP
- 混合节点架构
- Karpenter 弹性伸缩
- Terraform 示例
- 迁移到社区 Ingress
- AGS Browser Agent
- AGS Data Analysis
- AGS HTML Processing
- AGS Mini RL
- AGS Mobile Use
- AGS Shop Assistant

每个条目都尽量补上来源仓库、使用场景、标签、耗时、风险等级和执行提示。列表页先解决"我应该选哪个 Recipe"，详情页再解决"我怎么安全地执行它"。

这一步做完以后，Cookbook 不再是文档的附属品，而是 Workshop 的任务入口。

## 旧逻辑清理：不要让项目里存在两套真相

迁移完成以后，我又问了一个很现实的问题：项目里的 MkDocs 逻辑还有用吗？

如果新版已经完全承接旧文档和 Cookbook，那继续保留 MkDocs 配置、旧 `docs/` 目录、`overrides/`、旧 GitHub Actions 和 Python 文档依赖，其实会带来两个问题：

第一，后续维护者不知道该改哪套。新站点和旧站点同时存在，就很容易出现"线上是新版，仓库里还有旧版入口"的混乱。

第二，CI/CD 也会变复杂。部署失败时，很难判断到底是 Astro 的问题，还是遗留 MkDocs 流程的问题。

所以最后做了一次遗留清理：删掉旧 MkDocs 管线，更新 README、CODEBUDDY、PR 模板、CODEOWNERS 和相关链接，让项目只剩一套主体验。

这个动作看起来像 housekeeping，但对长期维护很重要。迁移项目最怕的不是新东西做不出来，而是旧东西半死不活地留在旁边。

## 真实浏览器比想象更诚实

这次还有一个很重要的工作方式：用真实浏览器体验。

我让 Codex 打开线上站点和本地站点，不只是看构建是否成功，而是像用户一样走一遍：首页、导航、Cookbook 列表、Recipe 详情、移动端布局、按钮、回链、旧入口。

很多问题只有在浏览器里才会变得明显。比如：

- 侧边栏一级目录太长，信息层级不清楚。
- Cookbook 卡片在某些尺寸下堆叠。
- 旧版 Cookbook 入口迁移后还残留，容易让用户走回旧路径。
- 详情页如果只有说明，没有风险、费用、参数、清理步骤，执行感还是不完整。

这些问题很难靠"代码看起来没错"发现。它们必须通过真实界面、截图和用户视角暴露出来。

这也让我更确定：Agent 参与产品构建时，浏览器体验应该是常规流程，而不是最后补一眼。

## 另一个会话：从内容工厂到文档编译 Agent

和这次 Workshop 重构并行的，还有另一个 Codex 会话。它一开始是在做"自主内容工厂 Agent"的设计，后来又往前走了一步：把站点文档变成像程序一样可以被审计、修复和验证的对象。

前半段的目标很直接：能不能让一个 Agent 系统每天自动产出一个内容 PR？这个 PR 里包含选题、调研报告、博客文章和审核结果。它不是直接发布内容，而是把内容推进到"可被人审阅"的状态。

设计里拆了四个 Agent：

- TopicPicker：从 TODO、ROADMAP 和已有文章里选择当天选题。
- Researcher：围绕选题做产品调研、竞品分析和用户需求整理。
- Writer：基于调研输出文章初稿。
- Reviewer：检查内容质量、格式、敏感信息和 Agent-First 规范，并创建 PR。

这套设计我觉得有两个关键点。

第一个关键点是职责分离。写作 Agent 不负责证明自己写得好，审核 Agent 也不负责重写一切。生成和审核分开，系统才不会陷入"自己给自己打满分"的幻觉。

第二个关键点是失败路径。一次本地测试里，流程其实跑通了：选题完成、调研完成、文章生成完成。但 Reviewer 没有放行，因为文章太短，并且缺少 API 参数表、操作示例、可执行验证命令等 Agent-First 要素。

这不是失败，反而是一个好信号。

因为它说明这个系统不是只会生成内容，也能识别内容还没有达到标准。更重要的是，它可以把"不合格的初稿 + 审核意见 + PR"交给人，而不是假装一切完成。

对我来说，这比"自动发布一篇文章"更有价值。真正可靠的 Agent 系统，不应该追求完全无人过问，而应该清楚知道哪些地方必须让人接管。

## 文档也应该可以"编译"

另一个更有意思的输入，是我后来提了一个问题：如果把站点上的操作文档变成可自动验证，像程序可以编译通过一样，会怎样？

这就从"内容生产"进入了"文档维护流水线"。

这个设计不再只关心文章写得好不好，而是关心一篇操作文档能不能被验证：

- 静态结构是否完整：frontmatter、导航、标题、链接、代码块是否合理。
- Dry Run 是否能过：命令、参数、依赖和输入输出是否能被 Agent 理解。
- Live Verify 是否安全：真实云资源验证是否有明确门禁、费用归属、清理动作和回滚路径。
- PR 和 issue 是否闭环：每个问题都能被追踪、修复、复核和关闭。

这里我最喜欢的一个比喻是：文档不只是 Markdown 文件，它也应该有自己的 compile pipeline。

程序编译不过，不能合入主干；操作文档如果缺少参数、验证命令、清理步骤，或者真实执行会留下资源，也不应该被视为"完成"。

## 从设计到运行：一篇文档一个 issue

这个会话后来不只是停在设计上，还开始实际跑流程。

我们把基础操作路径下的操作文档作为第一批对象，范围是 `src/content/docs/basics/**` 下除 `index.md` 外的文档。最终创建了 18 个单篇跟踪 issue，覆盖集群、节点、节点池、Service、Supernode、Workload 等基础操作。

这个粒度很关键。

如果把所有文档塞进一个大 issue，最后很难知道哪篇已经验证、哪篇还在排队、哪篇失败后需要修复。拆成"一篇文档一个 issue"以后，文档就变成了可以被调度的工作单元：

- Compile Sweep Agent 负责逐篇检查和修复。
- Doc Change Router 负责把变更送入后续流程。
- Doc Review Router 负责审阅和质量门禁。
- Live Verify Router 只处理已经满足条件的真实验证任务。
- Live Verify Agent 负责真实执行、报告结果和清理。

实际触发时，18 个 issue 先从 `backlog` 提到 `todo`，然后生成 18 个 direct task。排在最前面的开始运行，其余文档排队等待。这个状态很重要，因为它说明流程不是只创建了一堆待办，而是真的进入了 Agent 执行队列。

这让我对"文档可编译"这件事有了更具体的感受：不是写一个总控脚本扫全站，而是把每篇文档变成独立的、可追踪的、可以被 Agent 认领和推进的对象。

## 收尾逻辑：验证通过不等于流程完成

还有一个很典型的小细节：Live Verify 跑过以后，有些 issue 已经有了 `live-verify-passed`，但还保留着 `live-verify-required`，状态也没有完全到 `done`。

这说明"验证成功"和"流程完成"不是一回事。

如果不处理这个尾巴，系统下次扫描时可能会重复验证，或者让人误以为还有待执行动作。所以后来补了一段 closure-only 逻辑：

- Live Verify 成功且 cleanup 无遗留时，保留 `live-verify-passed`。
- 移除 `live-verify-required`、`live-verify-failed`、`cleanup-required`。
- 如果 PR 已合并，或者验证目标就是主干，把 issue 置为 `done`。
- 已经 `done + live-verify-passed` 的项，后续 Router 默认跳过，除非明确要求重新跑。

这段逻辑被写进了 Live Verify Agent 和 Live Verify Router，并用两个已通过但未收尾的 issue 做了验证：它们从"已通过但未收尾"变成了真正的 done。

这个细节对 Agent 系统很重要。很多自动化流程不是死在主流程，而是死在边界状态：已经完成一半、标签没清、状态没动、下次又被扫出来。一个可靠的 Agent 工作流必须设计收尾，而不是只设计开始。

## 三条线其实是同一件事

表面上看，Workshop 重构、自主内容工厂 Agent、文档编译 Agent 是三件事：一个是网站体验，一个是内容生产系统，一个是文档质量流水线。

但做完以后我发现，它们其实在解决同一个问题：如何让知识从"静态内容"变成"可执行、可验证、可协作的工作流"。

Workshop 是面向用户的前台。它要让用户快速找到任务，理解风险，复制命令，验证结果，清理资源。

内容工厂 Agent 是面向生产的后台。它要让选题、调研、写作、审核、提交 PR 变成一条稳定链路。

文档编译 Agent 是面向质量的后台。它要让每篇操作文档都有自己的 issue、状态、验证门禁、修复记录和收尾逻辑。

三者连接起来，就是一个很有意思的循环：

1. Workshop 提供结构化的内容和执行场景。
2. 内容工厂 Agent 可以基于这些结构生成新内容。
3. 文档编译 Agent 可以逐篇审计、修复、Dry Run 和 Live Verify。
4. 人通过 PR、issue 和浏览器体验做最后判断。
5. 反馈再回到 Workshop 的页面、数据结构和 Agent 规则里。

这比传统文档站更接近我想象中的 Agent-First 知识系统。

## 这次实践给我的几个结论

第一，AI 编程不是省掉产品判断。  
导航怎么分层、Cookbook 应该暴露哪些执行信息、旧入口要不要删，这些都不是模型能凭空替你决定的。AI 可以执行得很快，但方向仍然来自产品判断。

第二，迁移不是搬文件，而是迁移用户路径。  
旧内容不能丢，旧入口不能误导，新入口要能承接原来的使用习惯。否则技术上迁移完成了，用户体验上还是断的。

第三，Agent-First 不等于给 Agent 写一段 Prompt。  
它需要参数表、验证命令、风险说明、清理步骤、来源链接这些结构化上下文。没有这些，Agent 只能猜；有了这些，Agent 才能稳定执行。

第四，审核 Agent 可能比写作 Agent 更重要。  
生成内容已经不稀奇了，真正困难的是让系统知道什么内容不应该直接进入主线。那个"没有通过审核"的测试，反而让我对这套设计更有信心。

第五，文档也需要类似编译的门禁。  
对操作文档来说，能打开页面只是最低要求。真正有用的文档应该能通过静态检查、Dry Run、必要时的 Live Verify，并且能在验证后清理资源、关闭 issue。

第六，收尾逻辑是 Agent 工作流的一等公民。  
一个流程不能只定义怎么开始，还要定义什么叫完成、什么状态应该跳过、什么情况必须重新跑。`live-verify-passed` 之后还要移除 required 标签，就是一个很小但很关键的例子。

第七，浏览器体验是 AI 协作里的硬证据。  
构建通过只能说明代码能跑，页面好不好用还是要打开看。截图、真实交互、线上部署结果，都是让 AI 协作从"感觉完成"变成"真的可用"的关键。

## 最后

这次 Workshop 的更新，对我来说不只是一次站点改版。

它更像是一次工作方式的实验：我用 Codex 做持续协作，把一个旧文档站逐步改成更适合人和 Agent 一起使用的 Workshop；同时在另一个会话里设计自主内容工厂和文档编译流水线，让内容生产、质量审核、真实验证和 issue 收尾都进入可编排的流程。

这两条线合在一起，让我更清楚地看到一个方向：

未来的技术文档不应该只是"写给人看的说明书"。它应该同时是给人看的界面、给 Agent 用的上下文、给自动化系统执行的任务描述，以及给团队协作审阅的变更单元。

如果说上一阶段的 AI 编程是在证明"产品经理也能把想法做出来"，那这一次更像是在证明另一件事：

我们不只是可以用 AI 写代码，还可以用 AI 重新组织知识、流程和协作方式。
