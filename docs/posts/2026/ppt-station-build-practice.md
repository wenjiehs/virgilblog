# 给 AI 搭了一个 PPT 工厂：从"空文件夹"到"自动生产线"的一下午

## 🤔 为什么要做这件事

做产品经理久了，有一个逃不掉的宿命——**做 PPT**。

产品介绍 PPT、客户 Pitch PPT、内部汇报 PPT、售前方案 PPT……每个都要求"视觉统一、口径一致、数据脱敏"，每个都要花半天到一天时间。更痛苦的是，做完一版母模板后，还要派生 5 个子版本，每个子版本又要微调内容、换色、换 Logo。

之前我已经搭了好几个 AI 工作站——产品策划工作站、竞品分析工作站、文档维护工作站。它们都有一个共同特点：**给 AI 建立上下文 + 定义规则 + 让它在约束内自由发挥**。

那 PPT 能不能也这样？

我的想法很简单：**搭一个 PPT 工作站，我只提供大纲，AI 负责生成**。视觉规范、字体偏好、品牌色这些东西，工作站自己"记住"。

## 🎯 目标

一个下午，搭出一个能工作的 PPT 工作站：

1. **有清晰的目录结构**——工作站定义类文件和 PPT 业务文件分开
2. **有默认的视觉规范**——从已有 PPT 自动提取，不用手写
3. **有记忆能力**——记住我的偏好（字体、配色、布局习惯）
4. **能实际生成 PPTX**——不是 mockup，是可以直接用 PowerPoint 打开编辑的真文件
5. **大纲完整归档**——每次输入的原始大纲不能丢

## 🛠️ 实际动手过程

### 第一步：搭骨架（5 分钟）

告诉 AI 我的需求后，它很快给出了目录设计：

```
PPTStation/
├── config/styles/         # 视觉规范（定义类）
├── config/templates/      # 模板定义（定义类）
├── projects/              # PPT 项目（业务类）
│   └── [name]/
│       ├── drafts/        # 原始大纲（完整保留）
│       ├── outline.md     # 工作大纲
│       ├── output/        # 生成的 PPTX/HTML
│       └── assets/        # 素材
├── memory/                # 偏好记忆
├── CHANGELOG.md           # 工作站迭代记录
└── .codebuddy/rules/      # 工作站规则
```

**定义类和业务类严格分离**，这个设计我很满意。`CHANGELOG.md` 是后来加的——因为我知道这个工作站会持续迭代，需要有变更记录。

### 第二步：从已有 PPT 提取视觉规范（15 分钟）

这一步最有意思。我在云文档上有一份已有的 PPT，风格是我喜欢的。我让 AI：

1. 通过云文档 MCP 导出 PPTX
2. 用 `python-pptx` 解析主题 XML、幻灯片结构、字体、颜色
3. 提取所有图片素材
4. 自动生成 `config/styles/default.md`

提取出来的规范包括完整色板（主色、辅色、强调色等）、字体定义、各种页面类型的布局规则、甚至 CSS 变量——后面生成 HTML 版 PPT 时可以直接用。

> 说实话，如果让我手动整理这些规范，至少要半天。AI 用 python-pptx 解析 XML 后自动生成，15 分钟搞定。

### 第三步：第一份 PPT 生成（20 分钟）

我给了一份 20 页的某云原生产品介绍大纲，AI 用 `pptxgenjs`（Node.js 的 PPT 生成库）生成了实际的 PPTX。

过程中踩了几个坑：

- **中文引号在 JS 字符串里的转义问题**——中文的 `""` 在 JS 双引号字符串里会导致语法错误，最后统一替换成了 ASCII 等价物
- **pptxgenjs 不支持 8 位颜色**（带 alpha 通道）——`FFFFFF80` 这种半透明白色会报错，需要手动转成 6 位近似色
- **字体 fallback**——品牌字体不一定每台机器都有，需要设 Arial 作为 fallback

最终生成了一个 539KB 的 PPTX，20 页，封面斜切蓝色色块、内容页蓝色卡片、结束页白色方块装饰——和提取的视觉规范一致。

### 第四步：偏好迭代（2 分钟）

我说了一句"字体我喜欢中文无衬线和楷体的组合"，AI 立刻：

1. 修改生成脚本的字体常量
2. 重新生成 PPTX
3. 更新 `config/styles/default.md`
4. 更新 `memory/preferences.md`
5. 同步更新 AI 记忆

以后所有新项目都会默认用这个字体组合。**这就是工作站"越来越懂我"的核心机制**——双轨记忆（文件 + AI memory）。

### 第五步：第二份 PPT + 内容丰富（25 分钟）

接着我给了 25 页的 TKE 主品牌产品介绍大纲（母模板），同时提供了一份已有的 79 页面客版 PPT 作为内容参考。

AI 的做法是：

1. 用 `markitdown` 提取面客版 PPT 的全部文本内容
2. 用 `python-pptx` 提取图片素材和视觉参数
3. 将面客版的技术细节（Meta Cluster 架构、VPC-CNI 网络方案、TKE Insight 资源大盘、作业帮案例等）融入到新大纲的对应页面
4. 重新生成 PPTX

最终产出 567KB，比第一版丰富了 16%。每一页都有具体的技术描述，不再是空洞的标题。

### 第六步：补完流程（发现问题）

过程中发现了两个流程问题，立刻修正：

1. **原始大纲没有完整保存**——最初的 `outline.md` 只存了摘要。修正：增加 `drafts/` 目录，规则里写死"第一步必须保存原始大纲"
2. **目录结构说明没同步更新**——README 和 rules 里的目录结构描述不一致。修正：统一更新

这种"边用边发现问题边修"的模式，正是工作站迭代的核心。

## 💭 真实感受

### 什么做得好

- **视觉规范提取是最大惊喜**。从一个 PPTX 文件自动提取完整的色板、字体、布局规则，生成可直接引用的规范文档——这个能力在传统工作流里需要设计师花半天整理。
- **双轨记忆真的有用**。文件级记忆（`memory/preferences.md`）给人看，AI 记忆给 AI 看。两者互补，下次新建项目时 AI 直接知道我喜欢什么字体、什么配色。
- **大纲驱动的生产模式很自然**。我只需要写大纲（其实大纲也是 AI 在另一个工作站里帮我写的），然后说"生成"，就能拿到可用的 PPTX。

### 什么需要改进

- **生成的 PPT 还是偏"模板感"**。卡片布局、表格、金句——这些元素虽然整齐，但缺少真正的设计感。图片、图标、渐变这些视觉元素还需要后续加入。
- **没有 Speaker Notes**。大纲里定义了每页的讲稿要点，但当前生成脚本没有写入 PPT 的备注区。这是下一步要补的。
- **pptxgenjs 的表达力有限**。斜切色块、自由形状这些复杂图形效果，用代码生成还是不如设计师手工调。未来可能需要"模板 PPTX + 内容填充"的混合模式。

### AI 的角色

在这个过程中，AI 做了什么：

- 设计目录结构
- 解析 PPTX XML 提取视觉规范
- 下载导出已有文档
- 编写 Node.js 生成脚本
- 调试中文编码、颜色格式等各种坑
- 维护记忆和规则文件

**我做了什么**：提需求、给大纲、提偏好、发现流程问题。

这大概就是"Human-on-the-Loop"的一个缩影——我在回路上监督和决策，AI 在回路里执行和迭代。

## 🤔 一些思考

### PPT 的本质是什么

做了这个工作站后，我重新想了一下 PPT 的本质。它不是"做幻灯片"，它是**"把信息按照特定视觉语言组织，传递给特定受众"**。

这个过程可以拆成三层：

1. **内容层**——说什么（大纲）
2. **视觉层**——怎么呈现（规范）
3. **表达层**——怎么讲（讲稿）

工作站目前解决了前两层。第三层（Speaker Notes）是下一步。

### 工作站模式的复利

这是我搭的第 N 个 AI 工作站了。越做越觉得，**工作站的价值在于复利**：

- 第一次花时间建规则、提取规范、定义流程
- 第二次直接复用，只需要给大纲
- 第三次连大纲都有模板了

TKE Housekeeper 的 PPT 是第一次生成，花了完整过程。TKE 主品牌的 PPT 是第二次，规范和脚手架都在了，快了很多。如果还有第三个子品牌（TKE Serverless），可能 10 分钟就搞定。

## 🎉 总结

一个下午，从空文件夹到能自动生产 PPT 的工作站。它还很粗糙——没有精美的动画、没有真正的设计感、Speaker Notes 也没写。但核心流程跑通了：

**大纲 → 视觉规范 → PPTX 生成 → 偏好记忆**

接下来要做的：
- 补 Speaker Notes 自动写入
- 支持"模板 PPTX + 内容替换"模式（更好的视觉效果）
- 精简版自动裁剪（大纲里定义了精简版保留哪些页）
- 接入腾讯文档 MCP，生成后直接上传

如果你也有类似的"重复性 PPT 制作"困扰，可以参考这个思路。核心不是某个具体工具，而是**"给 AI 建立上下文 + 定义约束 + 让它在约束内自由发挥"**这个模式。

## 🔄 续篇：三种方案的踩坑之旅，以及最终答案（2026-04-29 更新）

上面写的是"一个下午"的故事。但真正的故事才刚开始——后面我又花了大半天时间，**把三种 PPT 生成方案全部试了一遍**，最终找到了正确打开方式。

### 第一次碰壁：模板替换方案

上面那版用 pptxgenjs 代码硬绘的 PPT，虽然内容完整，但说实话——**丑**。纯色块 + 文字，缺少设计感。

于是我想到：能不能用公司的正式 PPT 模板（腾讯 CSIG 商务模板，105 页），用代码把内容填进去？

听起来很完美，实际踩了一个大坑。

**python-pptx 不支持 slide 复制**。这不是 API 没暴露那么简单——是 PowerPoint 的 .pptx 格式（本质是个 ZIP 包，里面是一堆 XML）在设计上就没考虑过"从一个文件拷 slide 到另一个文件"这个操作。每个 slide 的图片、字体、颜色主题都通过 rId（关系 ID）指向 slideMaster 和 slideLayout，跨文件复制后这些引用全部断裂。

我试了这些方案，**全部失败**：

| 方案 | 结果 |
|------|------|
| `add_slide` + XML 替换 | PowerPoint 报文件损坏 ❌ |
| 每页独立保存再合并 | 105 页模板太大，duplicate name 爆炸 ❌ |
| OPC 底层 ZIP 操作合并 | 关系文件冲突，还是损坏 ❌ |
| 精简模板（105→11 页）再合并 | 文件能打开了，但图片全部丢失 ❌ |
| d_bergeron 的 `spTree.insert_element_before` | 图片丢失（rId 冲突）❌ |
| rId 重映射方案 | 终于不损坏了，图片也有了！✅ |

最后那个 rId 重映射方案确实让文件能打开、图片能显示了。但——**内容替换只有 85% 成功**。因为 CSIG 模板里很多 shape 都叫同一个名字（"标题文本"），必须靠位置来区分。我加了 `role_nth` 匹配模式（按字号分类 + 位置排序），把成功率提到了 96%。

> 但说实话，为了那剩下的 4%，我写了 400+ 行引擎代码、debug 了无数次 XML。这条路的投入产出比太低了。

### 第二次碰壁：pptxgenjs + CSIG 视觉规范

回到代码硬绘方案，但这次我把 CSIG 模板的视觉规范完整提取出来——深蓝背景 `#0052D9`、白色方块装饰、彩色左边框卡片、圆形图标占位——然后在 pptxgenjs 里模拟这些效果。

效果比纯色块好了不少，但用户（也就是我自己）的反馈是：**"还是不够好"**。

代码硬绘的根本问题在于：你写的是坐标和颜色值，不是设计。每个 `addText({x: 1.2, y: 3.5, ...})` 都是在画布上"放砖块"，但真正的 PPT 设计是需要**留白、节奏、视觉重心**的。代码很难表达这些。

### 转折点：调研 NotebookLM 和开源方案

这时候我停下来，认真做了一轮调研。

**NotebookLM 的方案**很有意思：它用 AI（Nano Banana Pro 模型）直接"画"整页 slide 为一张图片。视觉效果确实好，但——**每页都是图片，文字不可编辑**。2026 年 2 月加了 PPTX 导出，但本质还是 PNG/JPEG 塞进 .pptx 容器。对企业场景来说，不可编辑是硬伤。

然后我在 GitHub Topics 里发现了 **PPT-MASTER**（8.9k stars），它的方案让我眼前一亮：

```
用户文档 → AI 分析 → 生成 SVG 布局 → 转换为 PowerPoint 原生 DrawingML → 可编辑 .pptx
```

关键在于 **SVG 中间层**：

1. AI 生成 SVG 代码（绝对坐标，精确到像素）
2. SVG 可以在浏览器里预览——先看效果再转 PPTX
3. SVG 到 DrawingML 的转换是确定性的（不像模板替换那样有匹配失败风险）
4. 每个 `<text>` 都变成 PowerPoint 的原生文本框，完全可编辑
5. 不需要模板文件，不涉及 slide 复制——前面踩的所有坑直接绕过

### PPT-MASTER 集成

集成过程意外顺利：

```bash
git clone https://github.com/hugohe3/ppt-master.git
cp -r skills/ppt-master .codebuddy/skills/
pip install python-pptx svglib reportlab Pillow numpy
```

然后用它的标准管线跑了一遍 TKE 产品介绍（25 页）：

1. `project_manager.py init` → 创建项目结构
2. **Strategist Phase** → AI 分析大纲，生成设计规范（配色/字体/布局节奏）
3. **Executor Phase** → 逐页生成 SVG 到 `svg_output/`
4. `finalize_svg.py` → 后处理（图标嵌入、圆角转 path）
5. `svg_to_pptx.py -s final` → 导出原生可编辑 PPTX

结果：

| 指标 | 模板替换方案 | pptxgenjs 方案 | PPT-MASTER 方案 |
|------|:---:|:---:|:---:|
| 文件大小 | 2.5 MB | 385 KB | **91 KB** |
| 总 shapes | ~200（很多是模板残留） | ~300 | **688** |
| 可编辑文本框 | 不稳定（85%~96%） | 100% | **100%（382 个）** |
| 文件损坏风险 | 高 | 零 | **零** |
| 视觉效果 | 模板原生 ✅ | 一般 | **好（AI 精确控制布局）** |
| 开发维护成本 | 高（400+ 行引擎） | 中 | **低（Skill 即用）** |

**91 KB，688 个原生 shapes，382 个可编辑文本框，零损坏**。每个文字都能在 PowerPoint 里直接点击修改。

### 为什么 SVG 中间层是正确答案

回头看，三种方案的本质区别在于**谁负责布局**：

| 方案 | 布局者 | 问题 |
|------|--------|------|
| 模板替换 | 设计师（模板）+ AI（填内容） | 填充时 shape 匹配失败，引擎代码越写越复杂 |
| 代码硬绘 | AI（写坐标） | AI 写的是"放砖块"，不是"做设计" |
| **SVG 中间层** | **AI（生成完整视觉页面）** | AI 对 SVG 的控制力远强于对 PPTX API 的控制力 |

SVG 是 AI 最擅长的输出格式之一——它是结构化的 XML，坐标精确，可预览可验证。让 AI 生成 SVG 再转 PPTX，比让 AI 直接操作 PPTX API 要可靠得多。

### 流程沉淀

最终我把整个流程固化成了 `ppt-generator` Agent V3.1：

```
用户大纲 → 归档 → 搜索资料丰富内容 → PPT-MASTER Skill 接管
                                        ↓
            Strategist (8 项确认) → Executor (逐页 SVG) → 导出 PPTX
```

**Step 2（搜索资料）被我标为最重要的一步**——PPT-MASTER 的 Executor 只负责视觉，内容完全取决于你给它的素材。所以我写了详细的 SOP：4 层资料来源（工作站→已有项目→联网搜索→官方文档），逐页内容准备标准（每个卡片 ≥ 30 字+具体数据），还接入了 3 个工作站作为资料源。

## 💭 更新后的思考

### 弯路不白走

模板替换方案虽然失败了，但让我彻底理解了 .pptx 的内部结构（ZIP 包 + OpenXML + Relationship）。这个认知在评估 PPT-MASTER 的 `svg_to_pptx.py` 时非常有用——我能看懂它在做什么，为什么它的方案不会像我之前的方案那样翻车。

### 工作站的"方案选型"也是迭代

这次经历让我更新了一个认知：**工作站不只是"功能迭代"，方案选型本身也是迭代**。

- v0.1：pptxgenjs 代码硬绘（能用但丑）
- v0.2-v0.4：python-pptx 模板替换（漂亮但不可靠）
- **v0.5：PPT-MASTER SVG 中间层（既好看又可靠）**

如果一开始就知道 PPT-MASTER，可以省下模板替换的 400 行引擎代码。但回头看，没有前面的踩坑，我可能不会理解"为什么 SVG 中间层是正确答案"——这需要先踩过"slide 复制会损坏文件"和"代码硬绘缺少设计感"这两个坑。

### 开源的力量

PPT-MASTER 是一个 8.9k stars 的开源项目，MIT 协议。它解决的问题（AI 生成原生可编辑 PPTX）和我踩过的坑完全吻合。这再次验证了一个原则：**在自己造轮子之前，先搜一搜有没有人已经造好了**。
